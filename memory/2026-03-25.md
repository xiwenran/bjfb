## 视频笔记支持

- 飞书 `视频封面` 字段中的第一张图片会作为视频封面候选
- 蚁小二官方 API 链路已接通视频发布参数：视频文件 + 封面图
- 比特浏览器小红书链路已接通视频发布页：
  - 使用 `target=video` 打开视频发布页
  - 上传视频
  - 再单独设置视频封面

## 小红书视频封面真实交互
- 真实页面不是直接显示“上传封面”按钮
- 正确流程是：
  1. 视频上传成功后出现“设置封面”区域
  2. 第一张默认封面悬停后显示“修改封面”
  3. 点击“修改封面”进入弹窗
  4. 在弹窗中点“上传图片”
  5. 选择本地封面图
  6. 点“确定”
- 比特浏览器自动化已按这条真实路径实现
- 封面图上传时优先寻找弹窗内真实图片 `input[type="file"]` 直接写入，避免先点击“上传图片”触发系统文件选择窗口后卡住，导致标题/正文/标签阶段无法继续
- 封面弹窗关闭后，必须再等待标题框与正文区重新回到可编辑状态，才能继续输入内容

## 素材顺序规则修正
- 旧逻辑用 `parseInt(fileName)` 排序，会误把：
  - `课程封面_11.png`
  - `爆款大字emoji封面 (3).png`
  这类文件排到前面
- 新规则：
  - 只有“文件名前缀就是数字”的素材参与数字排序，如 `0.png / 1.png / 2.png`
  - 其余素材保留飞书原始顺序
- 这样可以保证：
  - `0` 基本稳定在第一页
  - 带数字但不是数字开头的封面/背景图不会抢到首页

## 比特浏览器稳定性补强
- 发布页现在优先复用当前 profile 的首个可用页面，并主动关闭多余的小红书标签页，避免一次发布后堆出多页发布窗口
- 视频和图文会强制打开各自对应的发布模式，避免：
  - 视频误落到图文页
  - 图文误复用视频页
- 视频页标题和正文现在采用“输入后回读校验，不一致就强制回填”的策略，避免页面焦点漂移导致内容没有真正写进去
- 话题标签优先点击精确候选项，找不到精确候选时才回车提交，降低模糊匹配到错误话题的概率
- 视频封面上传完成后，不再等待内部编辑器 DOM 节点完全出现，而是以“标题/正文占位文案 + 发布按钮可见”作为继续条件，减少长时间停留后自动存草稿的问题
- 视频页标题/正文找不到标准输入框时，允许直接点击占位文案进入编辑态，再继续输入
- 话题没有精确候选时，宁可跳过该标签也不再直接回车模糊匹配，避免选错话题
- 视频正文优先直接写入主正文区域的大编辑器，不再依赖第一个泛化 `contenteditable`，避免只写进标题或错误区域
- 视频封面上传完成后，等待时间从长超时改为短轮询，并在“确定”按钮可点击后尽快确认，减少挂太久后自动存草稿
- 视频封面上传完成后，顶部可能出现“封面上传失败，请检查网络重试”的误报 toast；不能把这个 toast 当作最终失败，
  应以主封面预览图或封面缩略图是否发生变化作为真实成功信号
- 视频封面成功信号继续补强：
  - 不再只靠缩略图数量变化
  - 还会比较整个“设置封面”弹窗的图片/文本签名是否变化
  - 这样即使 XHS 误报 toast，只要弹窗实际内容已刷新，也继续后续流程
- 视频/图文的话题标签确认进一步收紧：
  - 不能只输入不确认
  - 必须点页面真实下拉候选中的精确匹配项
  - 不能用宽泛 `text=` 匹配去点正文里的同名文本，否则会出现“看起来输入了，实际没有确认”的问题
  - 点击候选后仍需校验正文中已插入 `[话题]`，否则视为未成功确认
- 视频页发布前校验已放宽：
  - 不再只认某个标题 input 的 value
  - 改为读取页面真实可见的标题/正文内容
  - 避免出现“标题、正文、标签都已写进去，但校验误判标题为空”的问题
- 视频页发布前校验再补一层：
  - 如果标题/正文输入框本身读取不稳定，就退回到整页可见文本
  - 只要页面文字中已经包含“预期标题”和“预期正文前缀片段”，就视为校验通过
  - 这是视频页最稳的做法，因为视频发布页经常把真实内容渲染到预览/容器中，而不是稳定保存在单一 input.value

## 从图文发布扩展到视频发布的完整经验总结（高价值可复用）
- 这次真正跑通，不是靠“猜更多选择器”，而是先接受一个事实：**图文页和视频页是两套不同交互模型**，不能复用同一套假设
- 一开始对小红书页面“标题、正文、话题、封面、上传状态”都不确定时，最有效的方法不是先写大而全逻辑，而是按阶段拆开验证：
  1. 进入正确发布页
  2. 打通素材上传
  3. 打通标题输入
  4. 打通正文输入
  5. 打通话题确认
  6. 打通发布前校验
  7. 最后再处理更自然的人工节奏
- 图文页最先跑通的关键经验：
  - 不要执着于 `text=上传图文` 这类文字节点，要优先找页面上真实可用的上传控件和图片 input
  - 图片上传是否成功，优先看缩略图数量/预览变化，而不是等待抽象“完成”状态
  - 话题必须是“点击话题按钮 → 输入关键词 → 点击精确候选 → 校验正文插入 `[话题]`”
- 视频页是在图文思路上扩展出来的，但必须额外加一层“封面交互”：
  - 上传视频成功
  - 等页面出现“设置封面”
  - 悬停第一张默认封面
  - 点击“修改封面”
  - 进入封面弹窗
  - 上传封面图
  - 点击“确定”
  - 等弹窗真正关闭
  - 再继续标题、正文、话题
- 这次最重要的认知转变：
  - **页面 toast、普通文案、按钮 enabled 状态，都不能直接当成功信号**
  - 真实成功信号应该来自“页面内容发生了变化”
  - 例如：
    - 缩略图数量变化
    - 主封面预览变化
    - 封面弹窗图片/文本签名变化
    - 页面可见标题/正文/标签已经出现
- 这套方法论后面完全可以迁移到：
  - 知乎 BitBrowser 发布
  - 抖音 BitBrowser 发布
  - 其他平台网页自动化发布
- 可复用的统一原则：
  - 先找“用户真实会点击的控件”
  - 再找“页面真实发生变化的成功信号”
  - 最后才做“本地校验和节奏拟人化”
- 结论：
  - ClaudeCode 容易卡在“抽象理解页面结构”
  - 真正能跑通这种网页自动化的关键，是把页面当成一个**状态机**来拆，而不是当成几个静态选择器来猜

## 平台状态重发规则收紧
- 现在“是否允许再次发布”只看飞书里的平台状态字段
- 只有平台状态精确等于 `待发布` 时，该平台才会进入调度和实际发布
- `发布失败` 记录不会因为定时任务或点击“立即发布”而自动再次提交
- 若用户希望重发失败记录，必须手动把对应平台状态改回 `待发布`
- 前端列表会把 `待发布`、`非待发布`、`已完成` 分开显示，避免把失败记录继续当作待发布内容

## 定时区间改为扫描窗口（2026-03-26）
- 时间段配置保留，但不再是“到点轮询直接发布”，而是“在这些时间段内扫描并补建任务”
- 有发布时间且平台状态为 `待发布` 的记录：扫描到后立即创建精准到分钟的一次性本地任务
- 到记录自己的发布时间时，由本地任务直接执行发布，不再依赖 15 分钟/30 分钟间隔去碰到
- 没有发布时间的记录：不参与自动定时，只允许点击“立即发布”时人工提交
- “立即发布”也收紧为只发送平台状态为 `待发布`，且“无发布时间”或“发布时间已到”的记录

## 视频封面弹窗卡死问题 — 深度修复（2026-03-25 v2，最终版）

### 问题描述
- 封面图上传后弹窗停留接近 1 分钟，代码不点”确定”；手动点确定后标题/正文也不写入

### 根因（4 处，逐层排查）

**根因 1：`waitForCoverUpload` 完成信号错误**
- XiaoHongShu “确定”按钮**无论上传完没完，始终不会 disabled**
- 靠”确定可点击”判断上传完成完全无效，函数几乎立刻返回（上传还没完）

**根因 2：`humanClick` 被 modal 静默拦截**
- `humanClick` 用 `page.mouse.down/up` 发鼠标事件
- modal overlay 可以静默拦截这些事件，不抛错，Playwright 以为成功了
- `.catch()` 备选的 `trigger.click({ force: true })` 永远不触发

**根因 3：弹窗关闭检测 `!body.includes('设置封面')` 永远为 false**
- 主页面本身就有”设置封面”节标题文字
- 所有重试循环全部跑完 5 次（≈35s），两个兜底 waitForFunction 又各超时 20s+15s
- 总卡死约 70s

**根因 4：fillTitle/fillDescription 在弹窗未关时被调用**
- 弹窗背后的输入框 CSS 未隐藏（isVisible 返回 true）
- 但鼠标点击被 modal 截获，内容打到弹窗里而非标题框

### 最终修复方案

1. **`waitForCoverUpload`**：通过”取消”按钮定位弹窗容器，轮询检测容器内”上传中”文字与 loading 类元素消失，timeout 40s

2. **”确定”点击**：双保险——`trigger.click({ force: true })` + `page.evaluate(() => btn.click())`，每次循环都两步走，不依赖 humanClick

3. **弹窗关闭信号**：改为检查”取消”按钮是否消失（”取消”只在弹窗内，主页面不存在）

4. **`uploadVideoCover` 末尾**：再次确认”取消”不可见才返回，确保 fillTitle 时弹窗已关闭

5. **`fillTitle` / `fillDescription`**：增加 JS 直注入兜底——扫描页面最小可见 input（标题）和最大 contenteditable（正文），强制注入内容

### 修改文件
- `src/bitbrowser-xhs.js`：`waitForCoverUpload`、`uploadVideoCover` 确认段、`fillTitle`、`fillDescription`

⚠️ **关键经验**：
- 不能用页面文字判断弹窗状态，必须用弹窗**独有**元素（如”取消”按钮）
- XiaoHongShu modal 内的按钮对 `page.mouse` 事件有静默拦截，必须用 JS `btn.click()`
- 封面弹窗”确定”按钮无论何时都是 enabled 状态，不能作为上传完成信号
- ⚠️ **最重要**：绝不能直接 `setInputFiles` 到隐藏 input——会绕过 React 事件系统，文件附到 DOM 但服务端根本没收到，封面永远不会真正上传。必须先点击”上传图片”按钮触发 filechooser 事件，再通过 `chooser.setFiles()` 写入文件
- 上传完成最可靠信号：弹窗内**缩略图数量增加**（上传前先计数，上传后轮询直到数量增加）
