## 蚁小二发布规则
- 标题最大 50 字
- 正文最大 2000 字
- 云发布时 clientId 必须设为 null
- 本机发布时必须提供官方 clientId（设备号）
- 图文/视频/文章三种类型，字段要求不同
- 视频素材需包含 duration，封面需单独提供（用户自行提供，不依赖 ffmpeg）
- ⚠️ 抖音账号在蚁小二云端授权可能过期，需用户手动重新授权
- 官方 API Key 作为 Authorization 使用，当前项目不再依赖用户名密码登录链路
- ⚠️ **图片链接要求**：不能使用占位符链接（如via.placeholder.com），必须用真实在线图片或上传到蚁小二云存储

## 定时扫描窗口与独立任务
- 时间段配置保留，但职责已经调整为“扫描窗口”，不再直接承担“轮询到点就发”的职责
- 扫描窗口内，系统会检查飞书中平台状态精确等于 `待发布` 的记录
- 如果记录有明确发布时间、且还没有创建过独立任务，系统会立即补建一个精准到分钟的一次性本地任务
- 到记录自己的发布时间时，本地任务会直接执行发布，不再依赖扫描间隔去“碰到”
- 如果记录发布时间已经到了，但之前还没来得及建任务，扫描器会在本次扫描时立即补发
- 没有填写发布时间的记录，不参与自动定时；只有点击「立即发布」按钮时，才允许人工提交
- 因此：时间段负责“发现并补建任务”，发布时间负责“决定真实执行时刻”

- 飞书附件下载临时URL自带authcode，不需额外鉴权

- 蚁小二 API 路径: https://www.yixiaoer.cn/api
- 飞书附件 API 返回顺序可能不一致，需按文件名排序
- 飞书附件排序现在仅对“文件名前缀是数字”的素材做数字排序，其余保留原顺序
- 当前项目音乐搜索与模板配乐走官方 getAccountMusic / music 字段链路

- 持久化防重依赖三处：飞书平台状态、飞书备注成功标记、本地 `publish-ledger.json`

  - 不只看缩略图数量和主图 src

  - 图文与视频都要在点击候选后再校验正文里是否真正插入了 `[话题]`

  - 需要按页面真实可见标题/正文内容判断

- 视频页发布前校验进一步放宽：
  - 不再死盯某个标题 input 或正文编辑器
  - 只要页面可见文本里已经包含“预期标题”与“预期正文片段”，就允许继续发布
  - 这样可以避免视频页 DOM 切换导致“页面明明已经填好了，但本地校验误判失败”
- **BitBrowser 自动化通用方法论（可复用到知乎 / 抖音 / 其他平台）**
  - 一开始不要假设页面结构，先把链路拆成“进入发布页 → 上传素材 → 填标题 → 填正文 → 加标签/话题 → 点击发布 → 校验结果”
  - 每一步都优先找“页面真实可见、用户真的会点击的控件”，不要先依赖隐藏 input 或内部字段
  - 页面提示文字、toast、按钮 enabled 状态都不能直接当最终成功信号；要优先看“页面内容是否真实变化”
  - 上传成功优先看：缩略图数量变化、预览图变化、弹窗内容变化，而不是只看 toast
  - 输入成功优先看：页面最终显示出来的标题/正文/话题，而不是只看某个 input.value
  - 标签/话题必须走“输入 → 候选出现 → 精确点击确认 → 校验正文已插入”的完整闭环，不能只输入不确认
  - 弹窗流程必须用“弹窗独有元素”判断打开/关闭，不能用主页面也可能出现的普通文案
  - 调试顺序要按真实问题推进：先打通素材上传，再打通标题正文，再打通标签，再做发布前校验，最后再收敛成更自然的人类节奏
  - 一旦跑通，立即把“真实页面交互顺序、有效信号、无效信号、踩坑点”记进记忆文件，后续迁移新平台时直接复用方法，不从零猜 DOM
- ⚠️ **图片上传修复经验（2026-03-20）**：占位符图片导致发布失败，使用 https://source.unsplash.com/1080x1350/?keyword 作为高质量替代方案
- 若需重发某条记录，必须先把对应平台状态改回“待发布”；无需再手动清理本地账本

  - 再补更精致的空状态 / 预览状态 / 细节交互

- 2026-03-27 用户第二轮视觉反馈已确认：
  - 主色应收敛到白 / 灰 + 微信风绿色，不要再用偏花的渐变氛围色
  - 左栏一级导航只保留品牌和模块导航，不再放配置目录 / 数据目录
  - 中栏顶部要更统一，模块标题要更大、更像一级页面名
  - 中栏原则上只放“可直接操作”的内容；纯展示型内容优先放到右侧预览区
  - 设置页二级选项要做成更大的固定卡片，避免不同模块内容长度把结构打散
