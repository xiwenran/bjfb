## v2.5 — Skill 共享执行层收敛（🚧 进行中）

### 背景

`zhifa-upload` 与 `zhifa-pipeline` 已经共享同一套导入 API（`scan-folder` / `schedule` / `create-records` / `archive`），但执行层仍存在断层：多主题扫描合并、主题根目录封面下发、records 构造、上传后清空 `笔记主题` 并设置 `待处理`，仍需要主会话临时拼接 JSON 或补脚本。结果是 Skill 文本已经定义了全链路，实际执行却还会退化到会话级胶水。

### 目标

- 把 `zhifa-pipeline` 定义的正式流程下沉成共享执行层
- 让 `zhifa-upload` 成为“中途介入入口”，而不是另一套独立胶水流程
- 消除 `/tmp/zhifa_scan_result.json` 单文件覆盖、手动 `cp` 封面、手拼 `records JSON`、手动后处理这些会话级操作

### 涉及文件

- `scripts/skill_upload.py`
- `src/server.js`
- `skills/zhifa-upload/SKILL.md`
- `skills/zhifa-pipeline/SKILL.md`
- `docs/roadmap.md`

### 当前落地项

- `skill_upload.py` 新增：
  - `scan-many`
  - `materialize-covers`
  - `schedule`
  - `build-records`
  - `postprocess`
  - `archive`
- `src/server.js` 新增 `POST /api/import/postprocess`

### 验收标准

- 多主题上传任务不再依赖单一 `/tmp/zhifa_scan_result.json`
- 主题根目录封面可批量下发到模板目录，无需会话里手动 `cp`
- `records JSON` 可由扫描结果 + 调度结果 + 文案映射稳定生成
- 上传后处理通过正式接口完成：清空 `笔记主题`，并只对实际平台设置发布状态
- `zhifa-upload` / `zhifa-pipeline` 的执行命令统一指向共享 runner，而不是各自维护一套会话胶水
