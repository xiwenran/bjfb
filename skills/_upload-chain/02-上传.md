# 上传链路公共文档 · 02 上传（U4–U7）

> 前置：先读 `00-总纲.md`，完成 `01-上传前.md` 的 U1–U3。账号 / 店铺分组 / 覆盖策略的判定见 `00-总纲.md`「账号与店铺分组」。

## U4：排期 + 构建 records + dry-run 预检（必停点②）

从 U1/U2 产出的 `<scan.json>` 读取扫描结果，结合用户输入和 U3 生成的 `content.json`，统一通过共享执行层生成排期和 records。**禁止主会话用内嵌 Python / Node 临时脚本手拼账号、时间和 records。**

### 分配规则

**规则一：按店铺分组绑定账号池**
先读取 `~/Library/Application Support/Zhifa/accounts.json`。每个扫描分组先由用户确认对应的小红书店铺组，再取该店铺组的小红书账号池；抖音账号从独立抖音账号池分配（判定优先级见总纲）。禁止把不同内容分组混到一个全局账号池统一分配，否则会把内容发到错误账号。

**规则二：覆盖策略必选**
排期前必须让用户显式确认 `coverageStrategy`（`strict/严格覆盖`、`balanced/尽量覆盖`、`minimum/只保底发布`，见总纲）。默认只能推荐 `minimum/只保底发布`，不能静默套用。

**规则三：时间约束写入 plan**
用户给出的日期、时间窗、固定 `seed` 和同主题处理选择必须写进 `<plan.json>`。`<plan.json>.accountGroups` 的唯一服务端格式是 `{账号名: 店铺组名}`；即使 `accounts.json` 以「店铺组→账号列表」保存，构造计划时也必须反转为账号→店铺组，禁止把原嵌套结构直接传给服务端。用户只说「9 点左右」时统一解释为 `08:00—12:00`，不得擅自收窄。排期前先走主题检查：**只纳入功能上线后写入主题索引的记录；旧记录不回填、不推断。** 有冲突时一次性展示全部冲突并询问三种选择（`auto_space`、调整时间窗、`allow_conflicts`）；确认绑定本轮输入 fingerprint，输入变化后必须重问；没有有效确认不得生成 schedule。主题索引、飞书或发布历史读取异常时立即停止，不以空数据继续。

**361 分钟 + 分钟唯一硬约束**：同一平台同一账号任意相邻笔记固定至少间隔 361 分钟，同批所有任务的具体分钟全局唯一；两者都是服务端和上传前校验共同执行的硬约束，不接受人工放行。`allow_conflicts` 只放行同店铺跨账号同主题冲突，不能绕过 361 分钟或分钟唯一。`timeWindows` 由 `schedule` 交给服务端展开，按固定 `seed` 分层随机生成具体分钟；`build-records` 直接使用结果，不二次随机。

**进入排期前还必须确认固定 `seed`**（非空），`build-records --seed` 必须与 `plan.json` 一致。

### 执行

```bash
python3 ~/zhifa/scripts/skill_upload.py schedule <scan.json> <plan.json> --output <schedule.json>
python3 ~/zhifa/scripts/skill_upload.py build-records <scan.json> <schedule.json> <content.json> <records.json> --seed <与 plan.json 相同的 seed>
python3 ~/zhifa/scripts/skill_upload.py export-preview <records.json> <preview_dir>
```

`schedule` 返回 `{schedule, unscheduled, stats, constraints}`：`schedule` 每项 `{topic, noteKey, platform, account, publishTime}`，其中 `publishTime` 已是服务端生成的 `YYYY-MM-DD HH:MM` 具体分钟；若仍返回时间窗字符串，视为排期失败并停止，禁止交给 `build-records` 二次处理。

**天序号推算**：用户给发布日期范围（如「5.1–5.4」），起始日=第 1 天，依次递推；执行 `date "+%Y-%m-%d"` 辅助推算绝对日期。单天发布则所有账号在同一天按第 1 天逻辑处理。

### 上传前排期计划确认（CRITICAL — 必停点②，不可跳过）

构建 `<records.json>` 后、执行 create 前，必须向用户展示完整排期计划并等待用户明确说「确认」或「按这个上传」。不能只说总数，必须让用户能看出每个账号每天排了几篇。展示内容至少包含：

1. 总数：每个平台多少条、总共多少条
2. 日期范围：每个平台从哪天排到哪天
3. 首日账号覆盖表：首日每个平台、每个账号分别安排几篇
4. 每日账号覆盖表：按日期列出每个平台每个账号的篇数
5. 今日特殊限制解释：若用户说「今天小红书 2 篇 / 抖音 1 篇」，必须明确写出这是「每账号」还是「整个平台总共」，不确定时停下追问
6. 示例明细：至少列出首日全部记录（平台、账号、时间、标题）

**首日覆盖硬规则**：首日覆盖表中任何用户指定账号为 0 篇时，必须高亮并停下确认，不能继续上传。例外：用户明确说「某账号今天不排」。

**数量歧义默认解释**：当用户同时给出「每天每账号：小红书 N 篇 / 抖音 M 篇」，又说「今天小红书安排 X 篇、抖音安排 Y 篇」时，默认按「今天每账号 X/Y 篇」理解；只有用户明确说「今天整个平台总共 X 篇」，才按平台总量理解。

**跨平台素材复用**：小红书和抖音是独立发布平台，同一篇素材笔记可以各自生成一条发布记录；排期器只禁止同一平台内重复使用同一 `noteKey`，不得把小红书已用过的素材全局排除给抖音。

### dry-run 预检（构建后、上传前）

```bash
python3 ~/zhifa/scripts/skill_upload.py create <records.json> --dry-run
```

预检项（全部通过才允许上传）：
1. **JSON 格式**：records 数组非空，每条 record 必填字段齐全
2. **图片路径**：每条 record 的 images 里所有文件路径真实存在于磁盘
3. **模板多样性**：每个账号使用的模板种类充足，且不存在某账号模板过度集中
4. **时间约束**：同平台同账号相邻发布时间差至少 361 分钟，且同批具体分钟全局唯一
5. **无重复**：同平台内 noteKey 无重复、账号+发布时间无重复；跨平台允许复用同一 noteKey

任一项不通过 → 打印具体违规条目，**退出码非 0，不继续上传**。

> **dry-run 语义边界**：`--dry-run` 只代表上述 5 项结构校验通过；它**不检查标题是否符合 SYSTEM_PROMPT 三原则，也不判断标题吸引力**。不得把 dry-run 通过表述为「标题合格」或「内容质量通过」。

### 上传前封面完整性校验（CRITICAL）

执行 create 前，只对「本轮确定有 `0*` 封面源」的主题做完整性校验。遍历 scan JSON 里对应 folderPath，用以下命令统计**每个文件夹**的主序号 0 封面文件数，确认等于用户为该主题提供的封面张数（逐文件夹检查，不是总数）：

```bash
find "<folderPath>" -maxdepth 1 -type f \( -iname "0.jpg" -o -iname "0(*).jpg" -o -iname "0（*）.jpg" -o -iname "0.*.jpg" -o -iname "0.png" -o -iname "0(*).png" -o -iname "0（*）.png" -o -iname "0.*.png" \) | wc -l
```

有封面源但数量不一致则停下报告，不允许带错误封面数量的记录上传。目录链路里完全没有 `0*` 的图文素材，按「无单独封面」记录并继续，不要为了通过校验移动或复制非 `0*` 图片。（用 `find` 而非 `ls 0*.jpg`，避免括号文件名在 shell 中触发 glob 扩展异常。）

### records 字段来源

| 字段 | 来源 |
|------|------|
| `topic` | scan JSON 里的 `topic` 字段 |
| `topicOverride` | **留空 `""`**，不要填内容——知发会把此字段写入飞书「笔记主题」列，用户不需要该列有值 |
| `noteKey` | scan JSON 里每条 note 的 `noteKey` 字段 |
| `folderPath` | scan JSON 里每条 note 的 `folderPath` 字段（绝对路径） |
| `images` | scan JSON 里每条 note 的 `images` 数组（已含 name/path/size，**直接复用，不要重新构建**） |
| `videos` | scan JSON 里每条 note 的 `videos` 数组（已含 name/path/size）。有视频时知发自动识别为视频笔记：视频上传到飞书「素材」字段，封面图（文件名主序号=0）上传到「视频封面」字段，内容类型设为「视频」 |
| `xiaohongshuAccount` | 按分配原则确定的该条笔记对应账号，没有小红书账号则留空 `""` |
| `douyinAccount` | 用户提供的抖音账号名，没有则留空 `""` |
| `publishTime` | 按分配原则确定的该条笔记对应时间槽，格式 `YYYY-MM-DD HH:mm` |
| `xiaohongshuChannel` | 用户指定的渠道，**未指定时固定填 `"蚁小二"`** |
| `title` | Claude 生成的标题（字数 8-20 字，搜索关键词为主体，emoji 最多 1 个） |
| `description` | Claude 生成的正文（首行搜索词展开+中间逐行列内容，末行可选、不写完整性话术，禁止留空） |
| `tags` | Claude 生成的标签数组（每个带 `#` 前缀，如 `["#经典常谈", "#马说"]`） |

**平台字段硬规则**：同一条记录只能填写 `xiaohongshuAccount` 或 `douyinAccount` 其中一个。若同一篇笔记要发小红书和抖音，必须在 `records` 里写成两条记录：一条只填小红书账号，一条只填抖音账号。**禁止一条记录同时带两个平台账号。**

**records JSON 结构示例**（写入 `/tmp/zhifa_records.json`）：

```json
{
  "records": [
    {
      "topic": "课件A",
      "topicOverride": "",
      "noteKey": "课件A/1",
      "folderPath": "~/笔记制作输出/合成图/课件A/1",
      "images": [
        {"name": "0.jpg", "path": "~/笔记制作输出/合成图/课件A/1/0.jpg", "size": 98765},
        {"name": "1.jpg", "path": "~/笔记制作输出/合成图/课件A/1/1.jpg", "size": 123456}
      ],
      "videos": [],
      "xiaohongshuAccount": "小红书-测试账号",
      "douyinAccount": "",
      "publishTime": "2026-04-30 15:00",
      "xiaohongshuChannel": "蚁小二",
      "title": "📌三国草船借箭讲义，一页讲透省时间",
      "description": "📌统编版语文《草船借箭》整理\n📝梳理了诸葛亮借箭的关键情节与人物性格线索\n✅配套问题设计和课堂讨论点，可直接展示\n🔥新接班或衔接课备课，这份能省下不少时间",
      "tags": ["#历史故事", "#三国", "#战争智慧"]
    }
  ]
}
```

**视频笔记字段写入规则**：当 `videos` 数组非空时，知发自动识别为视频笔记——视频文件上传到飞书「素材」字段，`images` 中文件名主序号=0 的图（`0.jpg`）上传到飞书「视频封面」字段，内容类型设为「视频」。图文笔记（`videos` 为空）行为不变。

**注意**：`title` 字段预填时，知发跳过自身 AI 生成，直接使用 Claude 写的文案。

## U5：上传前冷眼审查（CRITICAL — 不可跳过、不可降级）

**事前定位（CRITICAL）**：主会话进入本 Step 前必须粘贴本节标题及第一句原文。写不出 = 视为没意识到这一步 = 不允许推进。

**用户已审 ≠ 子代理审，不得豁免**：用户预览审的是标题质量主观感受，子代理审的是模板分配 / 标题字数 / 标签合规 / 调度合理性 / 数据完整性等**结构性问题**。两者根本不是同一件事。「用户已审 + dry-run 已过」不构成豁免理由。

审查前先导出可搜索预览（若 U4 已导出可复用），便于按账号、时间、noteKey、标题抽查：

```bash
python3 ~/zhifa/scripts/skill_upload.py export-preview /tmp/zhifa_records.json /tmp/zhifa_upload_preview
```

**派独立审查子代理**做独立审查。Codex 端按 Echo 规则声明 `子代理：冷眼审查 → model: gpt-5.5`；Claude 端按对应模型规则执行。派单含：

```
目标：上传前独立审查 /tmp/zhifa_records.json
审查项（每项 ✅/❌/⚠️）：
1. 模板分配：共享 schedule/build-records 的模板分散是否正确？每账号模板种类是否足够、是否存在某账号模板过度集中？
2. 标题质量：是否以搜索关键词为主体、自然流畅表达（见 SYSTEM_PROMPT 三原则）？字数 8-20字？emoji 是否≤1个？同主题不同笔记角度是否不同？有无年级词 / 缩写课文名 / 编造属性等违规？
2.5 正文质量：是否撰写（非空）？是否按首行搜索词展开/中间列内容的结构写（末行可选，不强制）？有无「图片即为完整内容/完整内容见图/翻图查看/按图整理」等完整性话术（有即打回）？
3. 调度合理性：服务端分层随机是否覆盖用户窗口？同批分钟是否唯一？同平台同账号相邻笔记是否至少间隔 361 分钟？
4. 数据完整性：总数 = 账号数 × 每账号篇数？各账号篇数一致？
5. 封面完整性：存在 0* 封面源的主题，每个 folderPath 下主序号 0 封面数量是否一致；没有 0* 封面源的图文素材不因此阻断
报告格式：每项 ✅/❌/⚠️，❌ 项必须修复后才允许上传
派单末尾：审查结论写入 /tmp/agent-return-step-U5-<uuid>.txt（uuid 由子代理自己生成），返回该文件路径。
```

**子代理产出依赖锁（CRITICAL）**：主会话收到子代理返回后，**必须在回复中粘贴该文件的绝对路径**（不粘贴内容、不改写摘要、不写「已通过」四个字）。缺少文件路径 = 视为子代理未派 = 不允许推进 U6。理由：纯文本回报可被主会话「摘要为已通过」或伪造；落盘 + 引路径让用户可以 `cat` 文件原文核对，把「文本契约」变成「文件契约」。

**子代理不可用时的降级规则（Codex / Claude 等效）**：如果 Agent / spawn_agent / Task 工具不可用、额度不足、权限被拒或超时，不允许默认改成主会话自审后继续上传。必须先停下告知用户：

> 「上传前独立子代理审查未完成，原因：<具体原因>。是否允许降级为主会话本地结构审查并继续？请明确说『允许降级继续』。」

只有用户明确允许后，才能由主会话执行本地结构审查；审查结果仍必须写入 `/tmp/agent-return-step-U5-<uuid>.txt`，并在最终报告里标注「主会话本地审查，非独立子代理」。用户未明确允许时，不得进入 U6 上传。

审查通过 → 继续 U6 上传。❌ 不通过项必须修复后重审，修完重新跑 U4 dry-run + U5，不允许带错误上传。

## U6：上传建档（create）

U4 dry-run 预检 + U5 审查全部通过后，执行上传：

```bash
python3 ~/zhifa/scripts/skill_upload.py create /tmp/zhifa_records.json --output /tmp/zhifa_create_results.json
```

真实上传必须保留完整结果文件。若漏传 `--output`，脚本会自动写入 `/tmp/zhifa_upload_create_results_<YYYYMMDD-HHMMSS>.json` 并打印路径；后续 U7、U8、U9 都必须引用该结果文件，不得只看终端摘要。`skill_upload.py` 内置自适应分批，携带调度矩阵、标题、标签、images 路径。

- `images` 数组直接从 U1 scan 结果复用（含 size），不要重新生成
- `xiaohongshuChannel` 未指定时固定填 `"蚁小二"`
- 上传时发布状态字段留空（U8 后处理再根据用户意图补设「待处理」）

**飞书限流防护（≥50 条必读）**：飞书图片上传接口在密集请求时会限流，表现为 `upload_error` 逐批增多，最终连续全败。

| 总条数 | 每批条数 | 批间间隔 | 说明 |
|--------|---------|---------|------|
| ≤20 | 5 | 3s | 正常速度 |
| 21-50 | 3 | 5s | 适度降速 |
| 51-100 | 2 | 8s | 保守策略，每条记录含 10-15 张图片时飞书压力大 |
| >100 | 2 | 10s | 极保守 |

**断点续传（连续 3 批全败后）**：

```bash
python3 ~/zhifa/scripts/skill_upload.py create /tmp/zhifa_records.json --retry-failed --output /tmp/zhifa_retry_results.json
```

`--retry-failed` 自动从上次运行结果中提取失败的 noteKey，只重传失败的，不传全量。等待 60s 后再执行（让飞书限流窗口过去）。重复直到全部成功或用户放弃。

**防重复硬规则（CRITICAL — 宁可失败也不允许重复）**：
- 上传前 `--dry-run` 已检查同平台内 noteKey 和 账号+发布时间无重复
- 服务端指纹查重是最后一道防线：同一指纹的记录会被 skip，不会重复创建
- **超时或断开后绝对不能盲目重传全量**——必须先用 U7 验证方法确认哪些已成功，只传剩余的
- 如果无法确认服务端状态（日志也看不到、服务也连不上），**停下等用户决定**，不自动重试

## U7：上传完整性验证（verify）

**事前定位（CRITICAL）**：主会话进入本 Step 前必须粘贴本节标题及第一句原文。

上传完成后、后处理前，**必须验证全部 noteKey 都已成功写入飞书**。主路径用共享执行层 `verify` 子命令产出结构化凭据：

```bash
python3 ~/zhifa/scripts/skill_upload.py verify <create_results.json> --output /tmp/zhifa_verify_<时间戳>.json
```

**凭据字段强制粘贴数字**：verify 凭据 JSON 含 `actual_count`（实际成功数）、`status`（`ok` / `mismatch`）、`missing`（缺失的 recordId）。规则：
- **AI 禁止自报上传条数**，只能转述凭据 JSON 里的 `actual_count` 字段值
- 凭据 `status == "mismatch"` 时禁止报「全部成功」，必须如实列出 `missing` 字段里缺失的 `recordId`
- 不允许只写「已校验通过」四个字——必须把凭据数字贴到回复里让用户当场核对

**验证标准**：
- ✅ 全部 noteKey 上传成功（actual_count = 期望数，status 非 mismatch）
- ✅ 无重复（本轮内每个 noteKey 只成功 1 次）
- ✅ 各账号记录数一致（总数 ÷ 账号数 = 每账号篇数）

**验证失败 → 不继续后处理/归档**，先解决缺失或重复问题（走 U6 `--retry-failed` 补传）。

**降级（verify 不可用时）**：用本地导入日志核验——从 `~/Library/Caches/Zhifa/logs/import-debug.log` 提取本轮时间窗内 `createRecord 成功 noteKey=...` 行，与 `records.json` 期望 noteKey 集合对比，算出 `missing` / `dupes`，同样把原始数字粘贴到回复。**不要用 `/api/records`**（全量拉飞书记录，100+ 条时经常超时或返回空）。本地导入日志是服务端实时写的，`createRecord 成功` = 飞书确认写入，可信度等同。
