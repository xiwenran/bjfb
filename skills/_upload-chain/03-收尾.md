## U8：上传后强制后处理

**事前定位（CRITICAL）**：主会话进入本 Step 前必须粘贴本节标题及第一句原文。

**操作结果数字强制粘贴**：执行完 ① + ② 后必须在回复里写一行 `后处理完成：N 条记录已更新（清空笔记主题 + 设发布状态）`，N 是真实 updateRecord 调用成功次数。不允许只写「已后处理」——必须有数字。

上传成功后，**立即**对所有刚创建的记录执行以下两步，不等用户追问：

**① 清空笔记主题字段**：`create-records` 会把 `topic` 写入飞书「笔记主题」列。笔记主题的作用是触发 AI 写作自动扫描——标题已在 U3 预填的情况下，留着笔记主题会导致调度器自动覆盖 Claude 写的标题。必须在上传后立即清空。

**② 设置发布状态**：统一把 `小红书发布状态` 和/或 `抖音发布状态` 设为 `待处理`（哪个平台有账号就设哪个）。用户在飞书检查确认无误后，手动改为「待发布」触发调度器自动发布。

**执行方式**：优先使用共享执行层，不在会话里临时拼 FeishuClient 调用。

```bash
python3 ~/zhifa/scripts/skill_upload.py postprocess <records.json> <create_results.json> --batch-size 10 --output <postprocess_results.json>
python3 ~/zhifa/scripts/skill_upload.py summarize-postprocess <postprocess_results.json>
```

`create` 必须落完整上传结果 JSON，供 `postprocess`、完整性验证和回执核验复用；不得再临时手写脚本调用 `/api/import/create-records` 只为保存返回值。`postprocess` 必须只处理 `status === "success"` 且有 `recordId` 的创建结果，按 `--batch-size` 分批处理大批量（**默认 10 条一批**，与脚本默认一致，避免一次请求超时），落后处理结果 JSON；空平台不得写状态。完成后必须用 `summarize-postprocess` 汇总 updated、rows、平台状态、topic 空值和账号计数。若共享命令不可用，才允许把 FeishuClient 直连作为降级兜底，并必须向用户说明「本轮使用降级后处理，不是主路径」。

**反面教训（2026-05-14）**：上传 7 篇视频笔记后，笔记主题没清空 + 发布状态没设 + 标题没预填，用户逐一追问才补上。三个遗漏全是「上传后处理」缺位导致。

## U9：上传后回执核验（默认机械回执 + 异常触发审查）

**事前定位（CRITICAL）**：主会话进入本 Step 前必须粘贴本节标题及第一句原文。

**设计意图**：例行上传的正确性目标已被机械核验覆盖——U7 `verify` 凭据对账「创建数=期望数」，U8 `summarize-postprocess` 对账「后处理数、平台状态、空 topic、账号计数」。本 Step 的第一次上传前审查（U5）已由独立子代理审过输入数据的结构性问题。因此**例行上传后默认只做机械回执核验，不再派独立审查子代理**；只有机械核验暴露异常，或全链路唯一的那次独立审查缺席时，才升级为审查报告。

### 默认路径：机械回执核验（不派子代理）

主会话把 U7 `verify` 凭据 + U8 `summarize-postprocess` 两份**已有产物**拼成一张回执表，**粘贴原始数字**（不允许「已核验通过」四个字了事）：

| 核验项 | 数据来源 | 通过标准 |
|---|---|---|
| 创建完整性 | U7 verify 凭据 | `actual_count == 期望数`、`status != mismatch`、`missing` 为空 |
| 无重复 | U7 verify / 导入日志 | 本轮每个 noteKey 只成功 1 次 |
| 后处理 | U8 summarize | `updated == created`、topic 空值数 == created、平台状态全为「待处理」 |
| 账号分布 | U8 summarize | 各账号篇数一致，总数 ÷ 账号数为整数 |

四项全部通过 → 回执核验通过，进入 U10。

### 异常触发条款（命中任意一条，仍必须派独立审查子代理）

1. U7 verify `status == mismatch`，或 `missing` 非空
2. create 有失败项（`--retry-failed` 补传后仍有）
3. U8 `updated ≠ created`，或 postprocess 有失败项
4. 各账号篇数不一致
5. **U5 上传前审查被降级或未跑过**——拿不到 `/tmp/agent-return-step-U5-*.txt` 路径，或中途介入从 U6 起步。全链路只剩这一次独立审查，一旦缺席，本 Step 必须补上
6. 用户明确要求审查

触发时**派独立审查子代理**（确认飞书实际写入正确，审「实际结果」，与 U5 审「输入数据」不同）。Codex 端按 Echo 规则声明 `子代理：冷眼审查 → model: gpt-5.5`；Claude 端按对应模型规则执行。派单含：

```
目标：上传后独立审查实际写入飞书的结果
数据：<records.json> + <create_results.json> + <retry_results.json，如有> + <postprocess_results.json> + ~/Library/Caches/Zhifa/logs/import-debug.log（本轮时间窗）
审查项（每项 ✅/❌/⚠️）：
1. 上传完整性：本轮 createRecord 成功的 noteKey 数 = JSON 期望数？
2. 无重复：本轮无同 noteKey 被多次创建？
3. 各账号一致：每账号成功记录数 = 期望篇数？
4. 后处理：笔记主题已清空？发布状态已设「待处理」？
派单末尾：审查结论写入 /tmp/agent-return-step-U9-<uuid>.txt，返回该文件路径。
```

**子代理产出依赖锁**：触发审查时，主会话**必须粘贴子代理返回的文件绝对路径**（不粘贴内容、不改写摘要）。缺失 = 不允许进 U10。

**子代理不可用时的降级规则（Codex / Claude 等效）**：如果 Agent / spawn_agent / Task 工具不可用、额度不足、权限被拒或超时，不允许默认改成主会话自审后收尾。必须先停下告知用户：

> 「上传后独立子代理审查未完成，原因：<具体原因>。是否允许降级为主会话本地核验并继续收尾？请明确说『允许降级继续』。」

只有用户明确允许后，才能由主会话读取飞书实际记录做本地核验；核验结果仍必须写入 `/tmp/agent-return-step-U9-<uuid>.txt`，并在最终报告里标注「主会话本地审查，非独立子代理」。用户未明确允许时，不得进入 U10。

## U10：归档

**事前定位（CRITICAL）**：主会话进入本 Step 前必须粘贴本节标题及第一句原文。

**先按入口约定的「归档口径」二选一（或不归档）**：
- `归档未排期`（默认）→ 走 **6-A：归档未安排的笔记**
- `移出已排期` → 走 **6-B：移出已排期的笔记**
- `不归档` → 跳过本 Step，回复里写一行说明用户选了不归档

两个分支互斥，不要同时跑。都必须在**上传成功且 U7 verify 通过之后**才执行（源目录路径被 records.json 引用，先动会导致上传/补传读不到图）。

### 6-A：归档未安排的笔记（口径＝归档未排期）

**归档结果数字强制粘贴**：执行完 cp 后必须在回复里写一行 `已归档 N 篇笔记到 <路径>/_未安排备用/`，N 是 cp 实际复制的笔记数。不允许只写「已归档」。

上传完成后，**必须**对照本次发布计划检查：**有没有制作了但没排进发布的笔记？或用户在备注里单独标记「不发」的笔记？** 有的话归档到备用位置，便于以后补发。

**用户指定了归档位置 → 按用户指令执行**，不要用默认路径，也不要判断「已经在那了」——用户说移到 X，就移到 X，按字面执行。

**默认归档路径**：`<合成图根目录>/_未安排备用/`（在合成图同级建子文件夹）
**命名规则**：`{主题简称}-{原编号}-模板{融景模板号}/` —— 例：`屏-2-模板3/`、`致-9-模板17/`

```bash
mkdir -p "<合成图根目录>/_未安排备用"
# 对每篇未安排的笔记
cp -R "<合成图根目录>/<主题全称>/<原编号>" \
      "<合成图根目录>/_未安排备用/<主题简称>-<原编号>-模板<融景模板号>"
```

**写 README**：在备用文件夹根目录写 `README.md`：本期日期 + 项目名；表格列出每个文件夹的来源主题、原编号、模板号、为什么没排到；命名规则说明、文件结构示例、怎么用（如何拷回主目录走补发）。

**用户单独备注的「不发」笔记**：用户在对话里说「这几篇先不发」/「X 留着以后用」，也按同样规则放到 `_未安排备用/`，README 里另起一段说明「用户备注不发的」。

**补发时只拷出，不移回**：`_未安排备用/` 里的笔记需要补发时，从备用目录**拷出到新目录**再走发布流程，不要 mv 回原合成图目录——mv 会破坏备用存档，且合成图目录结构变化可能影响再次扫描。

**6-A 不触发归档**：已经全部排进发布计划的（无未安排）→ 跳过本分支，不建空文件夹；用户明确说「删掉不要」的笔记 → 直接 `rm -rf` 不留备用，省硬盘。

### 6-B：移出已排期的笔记（口径＝移出已排期）

用途：把**本轮排进飞书的笔记**从源目录移出，避免后续批次重复安排；**未排期的笔记留在源目录**，后续继续安排发布。这是 6-A 的反向操作。

**移出结果数字强制粘贴**：执行完 mv 后必须在回复里写一行 `已移出 N 篇已排期笔记到 <目标路径>/`，N 是实际 mv 的笔记数（= 本轮去重 noteKey 数）。不允许只写「已移出」。

**前置硬门槛**：必须在 create 上传成功 + U7 verify 通过 + 失败项补传完成之后才执行。上传/补传还没结束就 mv，会导致读不到源图。

**移动方式与目标路径**：默认 `mv`（真移走，源目录不再保留），默认目标 `<素材根目录>/_已排期移出/`（在源目录同级）。用户指定了路径或要求 `cp` 时按用户的来——`mv` 是不可逆写操作，入口的归档口径确认时必须让用户显式认可方式和路径。

**操作**：用共享执行层 `move-arranged` 命令，它读 `schedule.json` 的去重 noteKey，按 `scan.json` 的 `folderPath` 定位真实源文件夹（避免主题名全角/半角括号导致路径拼错），保留「素材根下相对层级」（如 `科目/主题/模板号`）本地 `mv` 到目标目录。先 `--dry-run` 干跑校验（源缺失 / 目标冲突则整体停止、不动任何文件），确认无误再真移：

```bash
python3 ~/zhifa/scripts/skill_upload.py move-arranged <素材根目录> <素材根目录>/_已排期移出 <schedule.json> <scan.json> --dry-run
python3 ~/zhifa/scripts/skill_upload.py move-arranged <素材根目录> <素材根目录>/_已排期移出 <schedule.json> <scan.json>
```

> ⚠️ 不要用 `archive` 命令做移出——`archive` 走服务端 `cpSync` 是**复制**、会**保留源目录**，达不到「移出避免重复排期」的目的；`move-arranged` 才是本地真移动。

命令内部已校验并在结束时报告：① 已移动数 = 去重 noteKey 数；② 源目录 0 残留（`os.path.isdir` 逐个复检）；③ 目标目录写 `_moved_manifest.json` 留档。主会话据此在回复里粘贴「已移出 N 篇」和残留核验结论。

**写 README**：在 `_已排期移出/` 根目录写 `README.md`：本期日期 + 项目名 + 上传条数 + verify 凭据结论；表格列出每个 noteKey、标题、发布安排（平台/账号/时间）；说明「目录结构与源目录一致，需补发时把模板子文件夹拷回源目录再走发布」「未排期笔记留在源目录后续继续安排」。

**补发时只拷出、不移回**：需要补发时从 `_已排期移出/` **拷出**到新目录再发，不要 mv 回源目录破坏存档。

**6-B 不触发移出**：用户选 `不归档`，或本轮无成功上传记录时，跳过本分支。

## U11：报告结果

**报告前硬门槛（CRITICAL — 不可跳过）**：在说「上传完成/已上传 N 条」之前，必须已在 U7 跑过 `verify` 凭据，并在报告里**粘贴 verify 凭据文件的绝对路径**。

- **AI 禁止自报上传条数**，只能转述 verify 凭据 JSON 里的 `actual_count` 字段值
- 凭据 `status == "mismatch"` 时禁止报「全部成功」，必须如实列出缺失的 `recordId`（见凭据 `missing` 字段）
- **缺这份凭据路径 = 不允许报告完成**

报告模板：

```
✅ 上传完成

处理 X 组，共 Z 篇笔记：
  ✓ 成功 Z 篇（来源：verify 凭据 actual_count，非 AI 自报）
  - 跳过 0 篇（已存在，指纹查重命中）
  ✗ 失败 0 篇

后处理：
  ✓ 笔记主题已清空
  ✓ 发布状态已设为「待处理」（用户在飞书确认后手动改「待发布」触发调度）

上传后回执核验：[通过（机械回执，见上方数字） / 触发审查（原因 + 审查报告文件路径）]

归档：[已安排 N 篇 → _已排期移出/ ｜ 未安排 M 篇 → _未安排备用/ ｜ 用户选不归档]

verify 凭据：/tmp/zhifa_verify_<时间戳>.json
```
