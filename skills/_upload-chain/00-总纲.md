# 上传链路公共文档 · 00 总纲

> **单一源头**：从「合成图已就绪」到「上传完成并归档」的全部规则，是 `zhifa-pipeline` 与 `zhifa-upload` 两个 skill **共用的唯一规范正文**。两个 SKILL.md 只保留各自入口和停顿点，链路细节一律以本目录为准；两处若出现分歧，以本目录为准，并在最终回复里指出分歧。
>
> **不要在两个 SKILL.md 里复制本目录的条款**——历史上正是各写一份导致同一个 `Step 5.5` 在两份文档里指相反的两件事、且 pipeline 漏掉了多条安全规则。改链路规则只改这里。

> **顺序说明（与旧版两份 SKILL 的两处理顺）**：① 上传前审查（U5）排在「构建 records」（U4）**之后**——旧 upload 把审查（Step4.5）放在构建 records（Step5）之前，但审查对象正是 records.json，属原文顺序缺陷，此处修正为「先构建再审」。② 上传后回执核验（U9）排在归档（U10）**之前**——把「飞书写入是否正确」这道数据闸门放在 U10 6-B `move-arranged`（不可逆 `mv` 移走源文件）之前，审出问题时源文件还在，便于补传。

**收尾必回（CRITICAL）**：完成上传后必须从当前步骤往后走完 U7 → U8 → U9 → U10 → U11，用户明确说「跳过某步」除外，但 AI 不得自己默认跳过。中途介入（用户手动做了一部分后接入）时，先判断当前进度对应哪个 U，从该 U 开始，往后把剩余步骤走完。

## 共享执行层命令（唯一合法执行入口）

所有链路步骤统一复用 `~/zhifa/scripts/skill_upload.py` 的共享子命令，不再用会话内临时脚本 / 手工 JSON 拼接替代：

```bash
python3 ~/zhifa/scripts/skill_upload.py scan <合成图文件夹>                                              # 单主题扫描
python3 ~/zhifa/scripts/skill_upload.py scan-many <topic_dir...> --output <scan.json>                    # 多主题合并扫描（多主题必用）
python3 ~/zhifa/scripts/skill_upload.py materialize-covers <scan.json>                                   # 把主题根/上级目录的 0* 封面下发到模板目录
python3 ~/zhifa/scripts/skill_upload.py schedule <scan.json> <plan.json> --output <schedule.json>        # 生成调度矩阵
python3 ~/zhifa/scripts/skill_upload.py build-records <scan.json> <schedule.json> <content.json> <records.json> --seed <YYYYMMDD-batch-name>
python3 ~/zhifa/scripts/skill_upload.py export-preview <records.json> <preview_dir>                      # 导出可搜索预览 md/csv
python3 ~/zhifa/scripts/skill_upload.py create <records.json> --output <create_results.json>             # 上传建档
python3 ~/zhifa/scripts/skill_upload.py verify <create_results.json> --output <verify.json>              # 上传完整性验证凭据（防谎报）
python3 ~/zhifa/scripts/skill_upload.py postprocess <records.json> <create_results.json> --batch-size 10 --output <postprocess_results.json>
python3 ~/zhifa/scripts/skill_upload.py summarize-postprocess <postprocess_results.json>                 # 汇总后处理结果
python3 ~/zhifa/scripts/skill_upload.py token-summary ...                                                # 见下「Token 节制」
python3 ~/zhifa/scripts/skill_upload.py archive <source_dir> <target_dir> <schedule.json>                # 服务端复制归档，保留源目录（U9 6-A）
python3 ~/zhifa/scripts/skill_upload.py move-arranged <source_dir> <target_dir> <schedule.json> <scan.json> [--dry-run]  # 本地 mv 移出已排期（U9 6-B）
```

规则：
- 不再使用会话内临时脚本或手工 JSON 拼接替代这些共享步骤；`zhifa-pipeline` 与 `zhifa-upload` 只在入口和停顿点上不同，底层执行层完全收敛到这套命令
- 服务端返回的 `constraints`（含 361 分钟、分钟唯一和 seed）必须原样传到 records；Python 不得覆盖或降级
- `--dry-run`、断点续传、结果合并、后处理、汇总、预览都由共享执行层输出，不在会话里手写 Node/Python 重试脚本

## Token 节制要求

进入 **U5 上传前审查** 和 **U9 上传后回执核验** 前，优先调用 `token-summary` 获取摘要，主会话只读摘要 JSON，不直接读完整 `records.json` / 上传结果 JSON：

```bash
# 上传前风险摘要（pre）
python3 ~/zhifa/scripts/skill_upload.py token-summary --records <records.json> --phase pre --output /tmp/zhifa_token_summary_pre.json

# 上传后结果摘要（post）
python3 ~/zhifa/scripts/skill_upload.py token-summary --create-results <create_results.json> --postprocess-results <postprocess_results.json> --phase post --output /tmp/zhifa_token_summary_post.json
```

- 主会话只读 `summary.json`（pre：record_count / platform_count / account_count / missing_media / duplicate_note_key / duplicate_account_time；post：created_count / create_failed_count / postprocessed_count / postprocess_failed_count）
- 只有 summary 标出具体异常时，才回读对应 records 局部
- 大批量（>20 条）records 不得整体读入主会话上下文
- **不得为节省 token 跳过 U5 CRITICAL 冷眼审查节点**；U9 默认走机械回执，但异常触发条款命中时同样不得跳过审查

## 前置检查：知发服务

每个涉及知发接口的步骤前检查服务：

```bash
curl -sf http://localhost:3210/api/import/preflight
```

返回 `{"ok":true}` 才继续。失败时**不要立刻判断「知发 App 未打开」**，先按顺序排查：

1. 查端口是否监听：`lsof -nP -iTCP:3210 -sTCP:LISTEN`
2. 查代理环境：`env | grep -i proxy`
3. 若命令环境报 `Operation not permitted`、`EPERM`，或请求被转到 `127.0.0.1:7897` 等代理端口，说明是当前执行环境 / 代理拦截，**不等于 App 未运行**

只有端口未监听，才提醒用户打开知发 App（macOS 菜单栏可见），等用户确认后继续。Codex / Claude 两端等效：都必须先做端口与代理排查，不得把本机 HTTP 被沙箱或代理拦截误报成「用户没开 App」。

## 账号与店铺分组（排期共用判定）

无论从 CHECKLIST（pipeline）还是一条消息 9 项确认（upload）收集参数，最终喂给 `schedule` 的账号必须按下述判定，不得把所有账号混进全局账号池。

**先读账号文件并输出给用户确认**：

```bash
cat ~/Library/Application\ Support/Zhifa/accounts.json
```

`accounts.json` 支持旧版平台平铺清单，也支持新版 `xiaohongshu.default` / `douyin.default` + `accountGroups` 店铺分组映射。店铺分组是**发布策略**，不是素材目录层级；小红书店铺组和抖音账号池分开配置，不得要求用户把素材再拆成小红书/抖音/账号目录。推荐结构：

```json
{
  "xiaohongshu": { "default": ["小淼老师", "晓晓老师", "小陈老师", "..."] },
  "douyin": { "default": ["小林老师", "叮当老师", "..."] },
  "accountGroups": {
    "小淼老师店铺": { "xiaohongshu": ["小淼老师", "晓晓老师", "小陈老师", "小婉老师"] },
    "可乐店铺": { "xiaohongshu": ["可乐", "拉面卷卷", "小刘老师", "橙子老师"] }
  }
}
```

用户在入口（CHECKLIST 或任务消息）已明确列了账号 / 店铺组时，以用户列的为准，跳过自动加载。

**优先级规则**：
1. 用户本次消息里明确指定的账号或店铺组，优先级最高
2. `~/Library/Application Support/Zhifa/accounts.json` 的 `accountGroups[小红书店铺组]` 命中时，作为默认推荐
3. 未命中时，只展示候选店铺组并追问，不得自动全选
4. 未命中 `accountGroups`，但存在旧版 `xiaohongshu` / `douyin` 平铺清单时，只能展示为候选账号，不得自动全选
5. 小红书店铺组只用于小红书；抖音使用独立抖音账号池，同一批素材可同步安排到抖音，并按独立账号池随机或轮转分配；若用户明确说某平台不发，再单独留空
6. 缺账号、缺店铺组、缺发布日期、缺时间段时，必须以选项形式追问用户，不得继续 U3/U4

**覆盖策略必确认（三选一，不得隐含默认）**：
- `strict` / `严格覆盖`：要求每个账号尽量覆盖全部主题，覆盖不全进入 `stats.violations`
- `balanced` / `尽量覆盖`：优先分散主题，允许部分账号不覆盖全主题，覆盖不全进入 `stats.warnings`
- `minimum` / `只保底发布`：只保证每个账号有内容发出，不要求覆盖全部主题

用户没说时默认**推荐** `minimum/只保底发布`，但仍必须显式问一句「本轮覆盖策略用哪一种？」，不能静默套用。

**账号变更时同步更新 accounts.json**：用户说「加个账号 XX」或「XX 不用了」，执行完任务后顺手更新 `accounts.json`，下次自动生效。

## 飞书记录红线（CRITICAL — 禁止删改已入库记录）

记录一旦通过 `skill_upload.py create` 写入飞书，即视为进入发布流程——用户可能已手动改状态为「待发布」并开始发布，**状态留空 ≠ 安全可删**。禁止建议删除、批量替换素材或清空已创建记录。如需修复内容问题，**AI 不得发起任何飞书 API 写入调用**，只能告知用户在飞书界面手动操作，不得提议程序化批量操作已有记录。

## 三模型分工

| 角色 | 职责 |
|---|---|
| 主会话 | 编排 U1–U11、判断异常、组织用户确认点、U9 机械回执核验、merge 与收尾 |
| 文本子代理 | U3 标题/正文/标签生成；Codex 端默认 `gpt-5.4-mini`，复杂内容可升 `gpt-5.4` |
| 冷眼审查子代理 | U5 上传前审查（必派）+ U9 异常触发时的审查；Codex 端按 Echo 规则声明 `子代理：冷眼审查 → model: gpt-5.5`，Claude 端按对应模型规则执行 |
| 代码 worker | 仅当共享执行层缺能力或脚本报错需改代码时才派；普通发布流程不得写临时脚本替代 runner |

## 失败处理规则

| 故障类型 | 处理方式 |
|---|---|
| 任何接口超时 / 失败 | 先 `GET /api/status` 读服务端真实状态，再决定下一步，不盲目重试 |
| 上传批次连续失败 3 次 | 终止，用 `create --retry-failed` 断点续传（等 60s 让限流窗口过去），附已传 / 未传清单 |
| 单批超时（≥1800s） | 先查导入日志确认实际写入了多少条，再决定是否补传 |
| client 断开 ≠ server 完成 | 断开后用 U7 的 verify / 本地导入日志确认实际写入了哪些 noteKey，再决定补传；不要用 `/api/records`（100+ 条会超时） |
| 服务崩溃（connection refused） | 检查服务进程，必要时重启知发 App，等 10s 再继续 |
| 无法确认服务端状态（日志看不到、服务连不上） | **停下等用户决定**，不自动重试 |
| 标题生成不通过自检 | fallback 调 `/api/ai-writing/generate`，失败则报告给用户手动填写 |
| 飞书字段不存在 | 不静默跳过，停下告知用户并提供三选一：我帮你建 / 你自己建 / 跳过这个字段 |
