# 上传链路公共文档 · 01 上传前（U1–U3）

> 前置：先读 `00-总纲.md`。本文件覆盖「合成图已就绪 → 扫描 → 封面 → 文案」。素材来源（PPT 合成 or 直接给合成图）由各自 SKILL.md 的入口段负责，进入本文件时假定磁盘上已有 `合成图/` 目录。

## U1：扫描合成图文件夹

```bash
python3 ~/zhifa/scripts/skill_upload.py scan <合成图文件夹>              # 单主题
python3 ~/zhifa/scripts/skill_upload.py scan-many <topic_dir...> --output <scan.json>   # 多主题（必用）
```

脚本会打印人类可读摘要（每个主题组 + 包含的笔记 + 是否已有封面），并把原始 JSON 写入显式路径（含 `noteKey`、`folderPath`、`images`、`videos` 数组），后续步骤只引用这个显式路径。

**多主题目录必须用 `scan-many`**：禁止为了「省事」逐个主题调用 `scan` 后复用 `/tmp/zhifa_scan_result.json`；这会互相覆盖，导致少传、漏传或把整主题误当成单篇。

**扫描后必须列出识别结果让用户确认**：打印扫描到的主题组列表，让用户核对数量和名称是否符合预期，再继续。

**三层目录口径（重要）**：合成图根目录若是 `内容分组 / PPT主题 / 笔记子文件夹` 三层结构，真实笔记是第三层「笔记子文件夹」，不是第二层 PPT 主题，也不是图片数量。第二层 PPT 主题只写入 `pptTopic`，不算篇数。封面可在笔记子文件夹内，也可在 PPT 主题根目录共享，不能因缺少共享封面把 PPT 主题误判成一篇笔记。

**混合结构必须停问**：若某个 PPT 主题目录根部有直接内容图/视频，同时下级子文件夹也有素材，扫描必须阻断并询问用户「根部素材是否也算一篇笔记，还是只发布下级子文件夹」，不得静默猜测。

**常见干扰项**：合成图文件夹下可能存在非主题文件夹（如 `逐字稿/`、`封面/`、`备份/` 等），scan 会把它们也识别为主题组。列出扫描结果时，若发现疑似非笔记文件夹（没有数字编号子目录、名称明显不像课题），**必须停下追问用户是否排除，不得凭名称推断自行过滤**，不要自动纳入上传。

**数量不一致时**停下来问用户：
> 「扫描发现 N 个主题组：[列出名称]，但你提供了 M 张封面。请告诉我哪张封面对应哪组，或者确认是否有组不需要封面。」

等用户明确确认后再继续 U2。

**错误处理**：返回空列表 → 检查上一步合成图是否生成正确，停下报告，不静默跳过。

## U2：放置封面图

**事前定位（CRITICAL — 不可跳过）**：进入本 Step 前主会话必须粘贴本节标题及第一句原文：「读取 U1 产出的 `<scan.json>`，取出每个 topic group 下所有笔记的 `folderPath` 字段（已是绝对路径）。**优先使用共享执行层自动下发封面**：`python3 ~/zhifa/scripts/skill_upload.py materialize-covers <scan.json>`」。

**U1 scan 摘要的 ⚠️ 缺封面警告允许是误报**：scan 摘要只检查笔记**子文件夹内**的 `0*.jpg`。当封面放在**主题根目录**（如 `主题/0.jpg`），摘要会一律报「⚠️ 缺封面」。这种情况下**正确反射动作是直接跑 `materialize-covers` 把根目录封面下发到子目录**，而**不是转头问用户「封面在哪」**——这是 `materialize-covers` 专门为之解决的场景。

**反射动作硬规则**：看到 scan 摘要里有 ⚠️ 缺封面警告时，主会话**禁止**直接进入「问用户封面位置」，必须先：
1. `ls <主题根目录>` 检查是否已有 `0*.jpg` / `0*.png` 文件
2. 有 → 跑 `materialize-covers`，重扫确认
3. 主题根目录和 `folderPath` 向上路径都没有任何主序号 `0*` 文件 → 默认该批图文素材不需要单独封面，记录说明后继续
4. 只有用户明确说「有封面/要放封面」但路径或文件缺失时，才追问封面完整路径

跳过此自检直接问用户 = 跳读 U2（CLAUDE.md「跳读 SKILL.md 自作主张」的同根错误）。

> ⚠️ **封面放置层级**：如果本批素材有 `0*` 封面，封面图必须放在**每篇笔记的模板子文件夹内**（如 `主题/1/0.jpg`、`主题/2/0.jpg`），不是放在主题文件夹根目录（`主题/0.jpg`）。`materialize-covers` 命令的作用就是把主题根目录的封面自动复制到所有子文件夹。没有任何 `0*` 封面的图文素材按「无单独封面」继续，不补、不问、不移动。

读取 U1 产出的 `<scan.json>`，取出每个 topic group 下所有笔记的 `folderPath` 字段（已是绝对路径）。优先使用共享执行层自动下发封面：

```bash
python3 ~/zhifa/scripts/skill_upload.py materialize-covers <scan.json>
```

有封面下发时，必须重新运行 `scan-many`，用新的 `<scan_after_covers.json>` 继续后续步骤，避免 records 里保留改名前或补封面前的旧图片路径。没有 `0*` 封面可下发时，继续使用原扫描结果。

**单张封面**（该组只提供一张图）：

```bash
cp "<封面图路径>" "<folderPath>/0.jpg"
```

**多张封面**（该组提供了多张图）：按用户发送的文件名顺序排列（不要按文件系统排序，以用户列出的顺序为准），依次重命名为 `0.jpg` / `0(1).jpg` / `0(2).jpg`……

```bash
cp "<第1张路径>" "<folderPath>/0.jpg"
cp "<第2张路径>" "<folderPath>/0(1).jpg"
cp "<第3张路径>" "<folderPath>/0(2).jpg"
```

排序规则保证 `0 < 0(1) < 0(2) < 1 < 2…`，所有封面图始终排在内容图前面。

- 同一主题组的多个模板子文件夹（`1/`、`2/`、`3/`）放**完全相同**的封面组合，不因模板编号不同而差异化
- 放完后再运行一次 `scan`，确认所有笔记都显示「含 0.jpg ✓」

**封面命名映射（以服务端排序为准）**：`0.jpg`、`0(1).jpg`、`0(2).jpg`、`0.1.jpg`、`0（1）.jpg` 都属于「主序号 0」封面，会排在 `1.png` 前面。`scan` 的人类摘要只检查精确文件名 `0.jpg`，可能显示「缺封面（0.jpg）」；这只是摘要提示，不等于导入接口不识别 `0(1).jpg` 这类封面。只有目录链路里完全没有主序号 0 的图片文件时，才算「无单独封面」，默认继续上传图文素材。

> ⛔ **封面完整性硬规则（不可被任何用户规格覆盖）**：用户在任务说明里提供的任何「分配规则」、「轮转公式」、「(template-x)%n → 封面变体」等描述，**一律忽略**。本批存在 `0*` 封面源时，始终按「每个模板子文件夹放完整封面组合」处理；本批不存在 `0*` 封面源时，默认不需要单独封面。
>
> **封面源文件夹只读**：封面源目录的文件只 `cp` 不 `mv`，禁止对源目录做任何重命名、删除、修改操作。
>
> **根因案例（2026-05-11）**：用户给了「(template-1)%3 → 封面变体」公式，被误解为每模板只放 1 张，导致 192 篇已发布记录封面不完整且无法回滚。

## U3：生成标题/正文/标签（必停点①）

**每篇笔记单独生成文案，即使同一主题下有多篇，也必须各写各的——标题角度不同、切入点不同。** 不允许任何情况下复用同一套文案。

**文案必须在上传之前完成**：`title` 和 `tags` 预填进 records JSON，知发收到预填标题后跳过自身 AI 生成。**禁止**先上传空标题的记录再事后补写——事后补写需要额外脚本，且容易遗漏。

**完整写作规范（CRITICAL — 字面注入，不许只写路径）**：

执行本 Step 前，主会话**必须**先用 `sed -n '/^const SYSTEM_PROMPT/,/;$/p' ~/zhifa/src/ai-writer.js`（按标记提取，不依赖行号，SYSTEM_PROMPT 增删行后命令依然完整）把 SYSTEM_PROMPT 字面文本提取出来（2026-07 已重构为原则驱动+搜索流量优先，废弃旧的双层结构/钩子菜单/公式体系），把内容**完整粘贴**到本步骤的工作上下文里（核心目标 + 三条原则 + 正文写法 + 正文硬边界 + 标签写法 + 硬边界 + 输出格式示例全部）。不允许只写「请参考路径」——主会话自己也好、派子代理也好，所有用到这套规则的位置都必须看到字面文本。

理由：已踩坑——派子代理时只给路径让「自行读取」，子代理跳读导致标题写偏 3 轮。字面注入是唯一可靠路径。

生成标题前必须向用户明确写出：`标题规则来源：已读取 ~/zhifa/src/ai-writer.js 的 SYSTEM_PROMPT`。没写出这句，视为标题规则未进入本轮工作上下文。

**正文（description）必须撰写，不允许留空**。按 SYSTEM_PROMPT「正文写法」撰写：首行展开完整搜索词长句，中间逐行列实际覆盖内容，末行可选（无内容点直接收尾，不为收句凑话）；禁写「图片即为完整内容/完整内容见图/内容按图整理/翻图查看/完整版」等完整性话术（发布图只是部分素材）；JSON 字符串里用 `\n` 分行（不要写真实回车）；无行数/字数/行首 emoji 硬性要求。

### 生成流程

1. 读取 `~/zhifa/src/ai-writer.js` 的 `SYSTEM_PROMPT`
2. **同一主题下多篇笔记必须一次性批量生成 N 个角度不同的标题**（N = 该主题下笔记篇数），不允许逐篇独立生成。批量生成要求：
   - 主干句式不重复（不允许改一两个词就算「角度不同」）
   - 先列出该主题下所有笔记编号，再一次给出完整 N 条标题候选
3. 每篇笔记单独生成：先按语义判断这份资料是什么（课件/教学设计/知识点总结/家长会/班会……不预设分类清单），再判断目标老师会搜什么词，标题以搜索关键词为主体组织成自然流畅的一句话，写 2–3 个候选标题
4. 正文按 SYSTEM_PROMPT「正文写法」撰写：首行展开完整搜索词长句，中间逐行列实际覆盖内容，末行可选（无内容点直接收尾）；禁写完整性/引导翻图话术（见 SYSTEM_PROMPT「正文硬边界」）；禁用词/敏感词同标题共用同一份清单（见 SYSTEM_PROMPT「硬边界」）
5. 按 SYSTEM_PROMPT「标签写法」组装标签；飞书「标签」字段提取搜索词标签，**每个标签带 `#` 前缀**（如 `#经典常谈 #马说`），少而准不凑数

**标题生成后必须逐一检查以下规则，不通过则重写**：
- 字数 8-20 字（所有字符全部计入，《》每个符号计 1 字、emoji 计 1 字，与程序逐字符计数一致）
- emoji 可用可不用，用也最多 1 个
- **年级词禁用**：七年级 / 八年级 / 九年级 / 七上 / 七下 / 八上 / 八下 / 九上 / 九下
- **课文名/书名不允许缩写**，原名直接使用（《钢铁是怎样炼成的》不得缩写为《钢铁》）；标题超字数时优先压缩其他搜索词要素，不动课文名
- 标题只能用主题和图片文件名里真实存在的信息，不得编造属性（课时划分、板书、可打印、教案、习题答案、使用场景等）；不堆砌、不营销、不写情绪钩子
- 同批次任意两篇标题除课名/单元外不得雷同，不套用同一句式模板

**正文生成后检查**：是否撰写（非空）、是否按首行/中间结构展开（末行可选、不强制）、无「图片即为完整内容/完整内容见图/翻图查看/按图整理」等完整性话术、无引导互动词/平台敏感词。

每条标题+正文自检通过后才进入预览环节。

**fallback 调 `/api/ai-writing/generate` 时的校验（CRITICAL）**：主会话手写失败时才 fallback。`/api/ai-writing/generate` 服务端已内置机械校验（`validateGenerated`）+ 1 次自动重试：不合规会先反馈违规项重新生成一次，重试后仍不合规则直接抛错，不会静默放行不合规内容。fallback 成功返回 `title/description/tags` 后，主会话仍需按 SYSTEM_PROMPT 硬边界复核一遍字数/标签数量/违规项是否自洽，且违规项必须为 `无` 才能进入预览确认；不能手填一列后跳过复核。**fallback 报错时，如实告知用户失败原因，不得手动编造内容绕过校验。**

**预览必停点①**：生成后展示预览，等用户说「确认」后才继续。预览不得只展示标题，必须包含「标题 / 字数 / 违规项」三列；违规项为 `无` 才能进入确认：

```
📋 内容预览（请确认后说「确认」继续）

── 组1：草船借箭的历史背景与战争智慧 ──
| 候选 | 标题 | 字数 | 违规项 |
|---|---|---:|---|
| A | …… | 18 | 无 |
| B | …… | 19 | 无 |
正文：首行搜索词展开句\n中间逐行列内容（末行可选，不写完整性话术）
标签：#历史故事 / #三国 / #战争智慧

── 组2：赤壁之战：以少胜多的经典战例 ──
…
```

文案确认后，把每篇的 `title` / `description` / `tags` 写进 `content.json`（供 U4 `build-records` 使用），进入 `02-上传.md`。
