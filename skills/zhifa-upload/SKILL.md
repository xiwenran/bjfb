# 融景合成图 → 知发上传 Skill

把融景已合成的笔记图，通过知发（zhifa，运行在 localhost:3210）上传到飞书多维表格，定时发布小红书 / 抖音。

## CLI 路径

```bash
python3 ~/zhifa/scripts/skill_upload.py
```

## 共享执行层（2026-06-04 起）

`zhifa-upload` 不再默认依赖会话内临时 JSON / 临时脚本。优先复用以下共享子命令：

```bash
python3 ~/zhifa/scripts/skill_upload.py scan-many <topic_dir...> --output <scan.json>
python3 ~/zhifa/scripts/skill_upload.py materialize-covers <scan.json>
python3 ~/zhifa/scripts/skill_upload.py schedule <scan.json> <plan.json> --output <schedule.json>
python3 ~/zhifa/scripts/skill_upload.py build-records <scan.json> <schedule.json> <content.json> <records.json> --seed <YYYYMMDD-batch-name>
python3 ~/zhifa/scripts/skill_upload.py create <records.json> --output <create_results.json>
python3 ~/zhifa/scripts/skill_upload.py postprocess <records.json> <create_results.json> --batch-size 50 --output <postprocess_results.json>
python3 ~/zhifa/scripts/skill_upload.py summarize-postprocess <postprocess_results.json>
python3 ~/zhifa/scripts/skill_upload.py export-preview <records.json> <preview_dir>
python3 ~/zhifa/scripts/skill_upload.py archive <source_dir> <target_dir> <schedule.json>
```

规则：
- 多主题上传优先用 `scan-many`，不要再用单一 `/tmp/zhifa_scan_result.json` 串行覆盖保存
- 主题根目录或上级目录已有主序号 `0*` 封面时，优先用 `materialize-covers` 批量下发到模板目录，不再在会话里逐条 `cp`
- 如果向上查找不到任何主序号 `0*` 封面，默认该批图文素材不需要单独封面；不要停下追问，也不要把非 `0*` 图片移动/复制进去冒充封面
- 账号排期必须按用户确认的店铺分组账号池执行，不得把所有账号混进全局账号池
- `schedule` 同时接受两类计划文件：底层原生 `accounts + timeSlots`，以及日常口径 `accounts + timeWindows`
- 排期前必须确认账号、账号→店铺组映射、日期、时间窗和非空 `seed`；用户只说“9 点左右”时统一解释为 `08:00—12:00`，不得擅自收窄
- `schedule` 先检查同店铺跨账号同主题冲突，一次性展示全部冲突并让用户选择 `auto_space`、调整时间窗或 `allow_conflicts`；确认绑定本轮输入 fingerprint，输入变化后必须重问
- `timeWindows` 由 `schedule` 交给服务端展开；服务端按固定 `seed` 分层随机生成具体分钟，同批分钟全局唯一；`build-records` 直接使用结果，不二次随机
- 同一平台同一账号任意相邻笔记固定至少间隔 361 分钟，这是不可放行硬约束；dry-run 和真实 create 都执行同一校验
- 主题检查只纳入功能上线后写入主题索引的记录；旧记录不回填、不推断。主题索引、飞书记录或发布历史读取异常时立即停止
- `records JSON` 由 `build-records` 生成，不再手工拼装
- 真实 `create` 必须保留 output；漏传 `--output` 时脚本会自动写 `/tmp/zhifa_upload_create_results_<YYYYMMDD-HHMMSS>.json` 并打印路径
- 上传后处理优先走 `postprocess --batch-size`，不要再在会话里临时拼 FeishuClient 调用
- 飞书限频补传、上传结果合并、后处理结果文件、后处理汇总和上传预览都应由共享执行层输出；不要再在会话里手写 Node/Python 重试脚本
- 服务端返回的 `constraints`（含 361 分钟、分钟唯一和 seed）必须原样传到 records；Python 不得覆盖或降级

## Token 节制要求（2026-06-15 起）

进入**上传前冷眼审查（Step 4.5）**和**上传后汇报（Step 6 / 6.8）**前，优先调用 `token-summary` 获取摘要，主会话只读摘要 JSON，不直接读完整 `records.json` / 上传结果 JSON：

```bash
# 上传前风险摘要（pre）
python3 ~/zhifa/scripts/skill_upload.py token-summary \
  --records <records.json> \
  --phase pre \
  --output /tmp/zhifa_token_summary_pre.json

# 上传后结果摘要（post）
python3 ~/zhifa/scripts/skill_upload.py token-summary \
  --create-results <create_results.json> \
  --postprocess-results <postprocess_results.json> \
  --phase post \
  --output /tmp/zhifa_token_summary_post.json
```

规则：
- 主会话只读 `summary.json`（pre：record_count / platform_count / account_count / missing_media / duplicate_note_key / duplicate_account_time；post：created_count / create_failed_count / postprocessed_count / postprocess_failed_count）
- 只有 summary 标出具体异常时，才回读对应 records 局部
- 大批量（>20 条）records 不得整体读入主会话上下文
- **不得为节省 token 跳过 Step 4.5 / Step 5.5 / Step 6 / Step 6.8 等 CRITICAL 冷眼审查节点**

### Step 1：检查知发服务

```bash
curl -sf http://localhost:3210/api/import/preflight
```

失败时不要立刻判断"知发 App 未打开"。先按顺序排查：

1. 查端口是否监听：`lsof -nP -iTCP:3210 -sTCP:LISTEN`
2. 查代理环境：`env | grep -i proxy`
3. 若命令环境报 `Operation not permitted`、`EPERM`，或请求被转到 `127.0.0.1:7897` 等代理端口，说明是当前执行环境/代理拦截，不等于 App 未运行

只有端口未监听，才提醒用户打开知发 App（macOS 菜单栏可见），等用户确认后继续。

**Codex / Claude 等效要求**：两边都必须先做上述端口与代理排查；不得把本机 HTTP 被沙箱或代理拦截，误报成"用户没开 App"。

### Step 2：扫描合成图文件夹

```bash
python3 ~/zhifa/scripts/skill_upload.py scan <合成图文件夹>
```

脚本会：
1. 打印人类可读摘要（每个主题组 + 包含的笔记 + 是否已有封面）
2. 把原始 JSON 写入稳定的扫描结果文件（例如 `/tmp/zhifa_scan_<date>.json` 或仓库内临时 `tmp_upload_scan_*.json`），Step 3、5 只引用这个显式路径

**多主题目录必须使用共享执行层**：

```bash
python3 ~/zhifa/scripts/skill_upload.py scan-many <topic_dir...> --output <scan.json>
```

禁止为了“省事”逐个主题调用 `scan` 后复用 `/tmp/zhifa_scan_result.json`；这会互相覆盖，导致少传、漏传或把整主题误当成单篇。

**扫描后必须列出识别结果让用户确认**：打印扫描到的主题组列表，让用户核对数量和名称是否符合预期，再继续。

**多主题目录的扫描方式（重要）**：合成图根目录下若是 `内容分组 / PPT主题 / 笔记子文件夹` 三层结构，真实笔记是第三层“笔记子文件夹”，不是第二层 PPT 主题，也不是图片数量。扫描器必须展开为 `内容分组 → PPT主题 → 笔记子文件夹`；封面可在笔记子文件夹内，也可在 PPT 主题根目录共享，不能因缺少共享封面把 PPT 主题误判为一篇笔记。

**混合结构必须停问**：若某个 PPT 主题目录根部有直接内容图/视频，同时下级子文件夹也有素材，扫描必须阻断并询问用户“根部素材是否也算一篇笔记，还是只发布下级子文件夹”，不得静默猜测。

**天序号推算**：用户给发布日期范围（如「5.1–5.4」），起始日=第1天，依次递推；执行 `date "+%Y-%m-%d"` 辅助推算绝对日期。单天发布则所有账号在同一天按规则一、二的第1天逻辑处理。

**上传前排期计划确认（CRITICAL — 不可跳过）**：构建 `/tmp/zhifa_records.json` 后、执行 create 前，必须向用户展示完整排期计划并等待用户明确说「确认」或「按这个上传」。不能只说总数，必须让用户能看出每个账号每天排了几篇。

展示内容至少包含：

1. 总数：每个平台多少条、总共多少条
2. 日期范围：每个平台从哪天排到哪天
3. 首日账号覆盖表：首日每个平台、每个账号分别安排几篇
4. 每日账号覆盖表：按日期列出每个平台每个账号的篇数
5. 今日特殊限制解释：若用户说"今天小红书 2 篇 / 抖音 1 篇"，必须明确写出这是"每账号"还是"整个平台总共"，不确定时停下追问
6. 示例明细：至少列出首日全部记录（平台、账号、时间、标题）

**首日覆盖硬规则**：首日覆盖表中任何用户指定账号为 0 篇时，必须高亮并停下确认，不能继续上传。例外：用户明确说"某账号今天不排"。

**数量歧义默认解释**：当用户同时给出"每天每账号：小红书 N 篇 / 抖音 M 篇"，又说"今天小红书安排 X 篇、抖音安排 Y 篇"时，默认按"今天每账号 X/Y 篇"理解；只有用户明确说"今天整个平台总共 X 篇"，才按平台总量理解。

**跨平台素材复用**：小红书和抖音是独立发布平台，同一篇素材笔记可以各自生成一条发布记录；排期器只禁止同一平台内重复使用同一 `noteKey`，不得把小红书已用过的素材全局排除给抖音。

**上传前封面完整性校验（CRITICAL）**：执行 create 前，只对"本轮确定有 `0*` 封面源"的主题做完整性校验。遍历 scan JSON 里对应 folderPath，用以下命令统计**每个文件夹**的主序号 0 封面文件数，确认等于用户为该主题提供的封面张数（逐文件夹检查，不是总数）：

```bash
find "<folderPath>" -maxdepth 1 -type f \( -iname "0.jpg" -o -iname "0(*).jpg" -o -iname "0（*）.jpg" -o -iname "0.*.jpg" -o -iname "0.png" -o -iname "0(*).png" -o -iname "0（*）.png" -o -iname "0.*.png" \) | wc -l
```

有封面源但数量不一致则停下报告，不允许带错误封面数量的记录上传。目录链路里完全没有 `0*` 的图文素材，按"无单独封面"记录并继续，不要为了通过校验移动或复制非 `0*` 图片。（用 `find` 而非 `ls 0*.jpg`，避免括号文件名在 shell 中触发 glob 扩展异常。）

```bash
python3 ~/zhifa/scripts/skill_upload.py create <records.json> --output <create_results.json>
```

真实上传必须保存完整结果文件。若漏传 `--output`，脚本会自动写入 `/tmp/zhifa_upload_create_results_<YYYYMMDD-HHMMSS>.json` 并打印路径；上传后审查和后处理必须引用该结果文件。

### Step 5.5：上传后强制后处理（CRITICAL — 每次上传后必执行）

**事前定位（CRITICAL）**：主会话进入本 Step 前必须粘贴本节标题及第一句原文。

**操作结果数字强制粘贴**：执行完 ① + ② 后必须在回复里写一行 `后处理完成：N 条记录已更新（清空笔记主题 + 设发布状态）`，N 是真实 updateRecord 调用成功次数。不允许只写"已后处理"——必须有数字。

上传成功后，**立即**对所有刚创建的记录执行以下两步，不等用户追问：

**① 清空笔记主题字段**：`create-records` 会把 `topic` 写入飞书「笔记主题」列。笔记主题的作用是触发 AI 写作自动扫描——标题已在 Step 4 预填的情况下，留着笔记主题会导致调度器自动覆盖 Claude 写的标题。必须在上传后立即清空。

**② 设置发布状态**：统一把 `小红书发布状态` 和/或 `抖音发布状态` 设为 `待处理`（哪个平台有账号就设哪个）。用户在飞书检查确认无误后，手动改为「待发布」触发调度器自动发布。

**执行方式**：优先使用共享执行层，不在会话里临时拼 FeishuClient 调用。

```bash
python3 ~/zhifa/scripts/skill_upload.py postprocess <records.json> <create_results.json> --batch-size 50 --output <postprocess_results.json>
python3 ~/zhifa/scripts/skill_upload.py summarize-postprocess <postprocess_results.json>
```

`create` 必须落完整上传结果 JSON，供 `postprocess`、上传完整性验证和冷眼审查复用；不得再临时手写脚本调用 `/api/import/create-records` 只为保存返回值。`postprocess` 必须只处理 `status === "success"` 且有 `recordId` 的创建结果，按 `--batch-size` 分批处理大批量，落后处理结果 JSON；空平台不得写状态。完成后必须用 `summarize-postprocess` 汇总 updated、rows、平台状态、topic 空值和账号计数。若共享命令不可用，才允许把 FeishuClient 直连作为降级兜底，并必须向用户说明“本轮使用降级后处理，不是主路径”。

**反面教训（2026-05-14）**：上传 7 篇视频笔记后，笔记主题没清空 + 发布状态没设 + 标题没预填，用户逐一追问才补上。三个遗漏全是"上传后处理"缺位导致。

**先按 Step 0 第 9 项「归档口径」二选一（或不归档）**：

#### 6-A：归档未安排的笔记（口径＝归档未排期）

**归档结果数字强制粘贴**：执行完 cp 后必须在回复里写一行 `已归档 N 篇笔记到 <路径>/_未安排备用/`，N 是 cp 实际复制的笔记数。不允许只写"已归档"。

**默认归档路径**：`<合成图根目录>/_未安排备用/`（在合成图同级建子文件夹）

```bash

# 对每篇未安排的笔记
cp -R "<合成图根目录>/<主题全称>/<原编号>" \
      "<合成图根目录>/_未安排备用/<主题简称>-<原编号>-模板<融景模板号>"
```

**写 README**：在备用文件夹根目录写一份 `README.md`：
- 本期日期 + 项目名
- 表格列出每个文件夹的来源主题、原编号、模板号、为什么没排到

**用户单独备注的"不发"笔记**：用户如果在对话里说"这几篇先不发" / "X 留着以后用"，也按同样规则放到 `_未安排备用/`，README 里另起一段说明"用户备注不发的"。

**补发时只拷出，不移回**：`_未安排备用/` 里的笔记需要补发时，从备用目录**拷出到新目录**再走发布流程，不要 mv 回原合成图目录——mv 会破坏备用存档，且合成图目录结构变化可能影响再次扫描。

#### 6-B：移出已排期的笔记（口径＝移出已排期）

**操作**：读本轮 `records.json`，对每个去重 `noteKey` 取 `folderPath`（源模板子文件夹绝对路径），保留「素材根下相对层级」（如 `科目/主题/模板号`）mv 到目标目录：

```bash
# 伪代码：对每个 noteKey 的 folderPath
# rel = folderPath 相对 <素材根目录> 的路径（如 数学课件/<主题>/<模板号>）
# mkdir -p <目标>/<rel 的父目录>；mv <folderPath> <目标>/<rel>
```

移完必须校验两条并在回复里报告：① 目标目录下模板子文件夹数 = N；② 源目录里这 N 个 folderPath 已 0 残留（`os.path.isdir` 逐个复检）。

**写 README**：在 `_已排期移出/` 根目录写 `README.md`：

**补发时只拷出、不移回**：需要补发时从 `_已排期移出/` **拷出**到新目录再发，不要 mv 回源目录破坏存档。

**6-B 不触发移出**：用户选 `不归档`，或本轮无成功上传记录时，跳过本分支。

**报告前硬门槛（CRITICAL — 不可跳过）**：

规则：
- **AI 禁止自报上传条数**，只能转述 verify 凭据 JSON 里的 `actual_count` 字段值
- 凭据 `status == "mismatch"` 时禁止报"全部成功"，必须如实列出缺失的 `recordId`（见凭据 `missing` 字段）
- **缺这份凭据路径 = 不允许报告完成**

```
✅ 上传完成

处理 2 组，共 4 篇笔记：
  ✓ 成功 4 篇（来源：verify 凭据 actual_count，非 AI 自报）
  - 跳过 0 篇（已存在，指纹查重命中）
  ✗ 失败 0 篇

后处理：
  ✓ 笔记主题已清空
  ✓ 发布状态已设为「待处理」（用户在飞书确认后手动改「待发布」触发调度）

```

## 与 zhifa-pipeline 的区分（入口判断标准）

**判断口诀：磁盘上能 `ls` 看到合成图（JPG/PNG）或视频文件（MP4/MOV）→ 用这个 Skill（zhifa-upload）；只有 PPT 文件还没合成 → 用 zhifa-pipeline。**

| 情况 | 正确入口 |
|------|---------|
| 合成图文件夹已存在（`ls` 能看到 .jpg/.png） | **zhifa-upload**（本 Skill） |
| 视频 + 封面已准备好（`ls` 能看到 .mp4/.mov + 0.jpg） | **zhifa-upload**（本 Skill） |
| 起点是 PPT，还没跑融景合成 | zhifa-pipeline |

**AI 强制自检**：接到"上传/发布/排期"类任务时，必须先执行 `ls <目标文件夹>` 确认素材文件是否存在，**不能凭用户口头描述或目录名推断**。看到图片/视频文件 → 选本 Skill；看不到图片（只有 PPT）→ 选 zhifa-pipeline。

## 注意事项

- 知发服务必须运行（macOS 菜单栏可见），否则 Step 1 会失败
- 查重：相同内容二次上传知发自动跳过（指纹查重），不会重复创建记录
- 封面图文件名无所谓，单张封面统一重命名为 `0.jpg`；多张封面依次命名为 `0.jpg`/`0(1).jpg`/`0(2).jpg`...，源文件原名不影响目标命名
- 多个模板共用同一封面组合：同一主题的 `1/`、`2/`、`3/` 文件夹放**完全相同**的封面组合，不因模板编号不同而差异化
- ⛔ **飞书记录红线（禁止删改已入库记录）**：记录一旦通过 `skill_upload.py create` 写入飞书，即视为进入发布流程——用户可能已手动改状态为"待发布"并开始发布，**状态留空 ≠ 安全可删**。禁止建议删除、批量替换素材或清空已创建记录。如需修复内容问题，**AI 不得发起任何飞书 API 写入调用**，只能告知用户在飞书界面手动操作，不得提议程序化批量操作已有记录。
- `images` 数组直接从 scan JSON 复用（含 size 字段），不要手动构建
- **发布状态统一设「待处理」（2026-05-15 调整）**：`create-records` 上传时发布状态留空（知发设计）。**Step 5.5 后处理统一设为「待处理」**，用户在飞书检查确认后手动改为「待发布」触发调度器自动发布。不再区分用户说"发布"还是"上传"——一律先待处理，人工确认后再发。
- **立即发布按钮也受人工闸门保护（设计意图，不是 bug）**：UI 发布页的「立即发布」按钮内部仍要求状态 `=== '待发布'` 才执行。上传后状态为空时点该按钮会被静默跳过——这是有意为之，避免文案/封面还没审核完就被人误触发。要立即发布某条，先去飞书把状态改成"待发布"再点按钮。
