# 融景合成图 → 知发上传 Skill

把融景已合成的笔记图，通过知发（zhifa，运行在 localhost:3210）上传到飞书多维表格，定时发布小红书 / 抖音。

## CLI 路径

```bash
python3 ~/zhifa/scripts/skill_upload.py
```

## 输入目录结构（融景标准输出）

```
合成图/
  课件A/          ← 主题文件夹（topic group）
    1/            ← 模板1（一篇笔记）
      1.jpg  2.jpg ...
    2/            ← 模板2（一篇笔记）
      1.jpg  2.jpg ...
  课件B/
    1/
      1.jpg  2.jpg ...
```

## 工作流程

### 收尾必回（CRITICAL）

规范同 zhifa-pipeline SKILL.md「收尾必回」章节。本 Skill 的收尾步骤为 **Step 5.5 → Step 6 → Step 7**，完成上传后不得跳过，用户明确说跳过除外。

### Step 0：确认参数

用户在**一条消息里**提供所有必要信息，之后只需在 Step 4 确认内容预览、Step 5 确认排期计划即可。

**标准输入格式：**

```
合成图文件夹：~/笔记制作输出/合成图
账号：小红书-账号A、小红书-账号B、小红书-账号C、小红书-账号D、小红书-账号E、小红书-账号F
发布日期：5.1–5.2
时间段：11:10 / 13:00 / 20:00

组1：~/Desktop/草船借箭封面.jpg
主题：草船借箭的历史背景与战争智慧

组2：~/Desktop/赤壁之战封面.jpg
主题：赤壁之战：以少胜多的经典战例
```

**账号清单与分组规则自动加载**：Step 0 开始时，先读取本地账号文件并输出给用户确认：

```bash
cat ~/Library/Application\ Support/Zhifa/accounts.json
```

`accounts.json` 支持旧版平台平铺清单，也支持新版 `accountGroups` 分组映射。账号分组是**发布策略**，不是素材目录层级；不得要求用户把素材再拆成小红书/抖音/账号目录。

推荐结构：

```json
{
  "xiaohongshu": ["芝士就是力量", "柿柿如意"],
  "douyin": ["芳芳老师", "小茜老师"],
  "accountGroups": {
    "教务资料": {
      "xiaohongshu": ["芝士就是力量", "柿柿如意"],
      "douyin": ["芳芳老师"]
    },
    "英语类": {
      "xiaohongshu": ["芝士就是力量"],
      "douyin": [],
      "douyinMode": "manual"
    },
    "综合类": {
      "xiaohongshu": ["柿柿如意"],
      "douyin": ["小茜老师"]
    }
  },
  "accountGroupAliases": {
    "教务类": "教务资料",
    "英语": "英语类",
    "中考语法": "英语类",
    "综合组": "综合类"
  }
}
```

输出格式（供用户复制修改）：

```
当前账号清单：
小红书（15 个）：芝士就是力量、柿柿如意、甜甜老师、可乐、拉面卷卷、浅浅、小雨老师、晓晓老师、文文老师、小苦瓜、小狗、苹果派、小陈老师、伊伊老师、小芽老师
抖音（7 个）：芳芳老师、小茜老师、珊珊学姐、小赵老师、小余老师、小晴老师、木子李老师
内容分组：
- 教务资料：小红书 芝士就是力量、柿柿如意；抖音 芳芳老师
- 英语类：小红书 小陈老师、晓晓老师；抖音 手动发布（不自动生成）
- 综合类：小红书 柿柿如意；抖音 小茜老师

请确认本次每个内容分组使用哪些账号（可直接复制修改）。如果没有提到账号、日期或时间段，必须停下追问，不能默认使用全部账号。
```

用户确认或修改后，以用户回复的分组账号规则 / 账号清单为准。如果用户在任务消息里已经明确列了账号，跳过自动加载，以用户列的为准。

**优先级规则**：
1. 用户本次消息里明确指定的账号 / 分组账号规则，优先级最高
2. `accounts.json.accountGroups[内容分组]` 命中时，作为默认推荐
3. 未精确命中时，先查 `accounts.json.accountGroupAliases[内容分组]`；例如 `教务类` → `教务资料`、`中考语法` → `英语类`
4. 别名仍未命中时，允许做规范化模糊匹配：去掉 `类/组/资料/素材` 等泛化后缀后再比较；只能在唯一命中时自动映射，多候选必须回问
5. 未命中 `accountGroups`，但存在旧版 `xiaohongshu` / `douyin` 平铺清单时，只能展示为候选账号，不得自动全选
6. 内容分组名包含 `英语`、`语法` 或 `中考语法` 时，默认归入 `英语类`；英语类抖音默认 `manual`，不自动生成抖音记录，除非用户明确说“这组也自动发抖音”
7. 缺账号、缺发布日期、缺时间段时，必须以选项形式追问用户，不得继续 Step 4/5

**账号变更时同步更新 accounts.json**：如果用户说"加个账号 XX"或"XX 不用了"，执行完任务后顺手更新 `accounts.json`，下次自动生效。

参数说明：
- **合成图文件夹**：融景输出的 `合成图/` 目录路径
- **封面图**：每个主题组一张，请提供**完整磁盘路径**；文件名无所谓，Skill 统一存为 `0.jpg`
- **主题**：每组一句话，Claude 用于写小红书文案
- **账号 / 分组账号规则**：一个或多个小红书/抖音账号，或按内容分组配置账号池。未指定时从 `accounts.json.accountGroups` 推荐；未命中分组时必须追问
- **发布日期**：支持「5.1–5.4」「后天到大后天」等自然语言；先执行 `date "+%Y-%m-%d"` 拿今天日期推算绝对日期；单天也可以直接写「后天」
- **时间段**：一个或多个发布时间（如「11:10 / 13:00 / 20:00」），多个时间段按规则二独立轮转；若用户指定了某账号的固定时间，以指定为准
- **发布渠道（xiaohongshuChannel）**：可选，默认 `蚁小二`；如果用其他渠道，用户需显式说明

**主题 ≥ 2 的发布任务（无论账号数量）必须一次性确认以下 6 项（缺任何一项不得开始）**：

| # | 参数 | 说明 |
|---|------|------|
| 1 | 分组账号规则 | 每个内容分组要发到哪些小红书/抖音账号，逐一列名；英语/语法分组是否跳过抖音 |
| 2 | 每个账号发布几个主题 | 全部主题？部分主题？ |
| 3 | 每天每账号发几篇 | 影响天数计算 |
| 4 | 起止日期 | 每个账号单独列，有差异的单独标注 |
| 5 | 小红书与抖音是否复用合成图 | 是 → 无需新模板；否 → 需额外模板 |
| 6 | 时间段 | 每个时段固定给哪个账号，或按轮转 |
| 7 | 封面是否已准备好 | 每个主题组的封面图路径是否已知；若未准备好，Step 3 会卡住 |

以上 7 项在用户第一条消息里全部确认，**执行过程中不再反复调整**。若用户消息缺项，先追问补全再继续。

**Step 0 封面参数识别**：读取用户参数时，若发现"按模板分配封面"、"轮转封面"、"(template-x)%n"、"封面变体"、"不同模板用不同封面"等描述，**直接忽略该规格，按默认处理**——每个模板子文件夹放完整封面组合，不询问用户。

**关于封面图的附件（@ 拖入）**：如果用户通过 @ 或拖拽发送封面图，先运行：
```bash
ls -la <用户说的路径或附件路径>
```
确认文件确实存在后再 cp。**如果路径不确定，直接问用户"封面图的完整路径是什么？"**，不要猜。

### Step 1：检查知发服务

```bash
curl -sf http://localhost:3210/api/import/preflight
```

失败时不要立刻判断"知发 App 未打开"。先按顺序排查：

1. 查端口是否监听：`lsof -nP -iTCP:3210 -sTCP:LISTEN`
2. 查代理环境：`env | grep -i proxy`
3. 若命令环境报 `Operation not permitted`、`EPERM`，或请求被转到 `127.0.0.1:7897` 等代理端口，说明是当前执行环境/代理拦截，不等于 App 未运行

只有端口未监听，才提醒用户打开知发 App（macOS 菜单栏可见），等用户确认后继续。

**Codex / Claude 等效要求**：两边都必须先做上述端口与代理排查；不得把本机 HTTP 被沙箱或代理拦截，误报成"用户没开 App"。

### Step 2：扫描合成图文件夹

```bash
python3 ~/zhifa/scripts/skill_upload.py scan <合成图文件夹>
```

脚本会：
1. 打印人类可读摘要（每个主题组 + 包含的笔记 + 是否已有封面）
2. 把原始 JSON 写入 `/tmp/zhifa_scan_result.json`（Step 3、5 会用到）

**扫描后必须列出识别结果让用户确认**：打印扫描到的主题组列表，让用户核对数量和名称是否符合预期，再继续。

**多主题目录的扫描方式（重要）**：合成图根目录下若有多个主题子文件夹（每个主题下再有模板编号子文件夹），**不要直接扫根目录**——扫描器会把每个主题文件夹当成一篇笔记，只扫出 N 个主题而非 N×M 篇笔记。正确做法是逐个主题文件夹单独扫描、合并 JSON 结果；或合成时就按「主题/模板/图片」结构组织，扫描才能一次到位。

**常见干扰项**：合成图文件夹下可能存在非主题文件夹（如 `逐字稿/`、`封面/`、`备份/` 等），scan 会把它们也识别为主题组。列出扫描结果时，若发现疑似非笔记文件夹（没有数字编号子目录、名称明显不像课题），**必须停下追问用户是否排除，不得凭名称推断自行过滤**，不要自动纳入上传。

**数量不一致时**停下来问用户：
> 「扫描发现 N 个主题组：[列出名称]，但你提供了 M 张封面。请告诉我哪张封面对应哪组，或者确认是否有组不需要封面。」

等用户明确确认后再继续 Step 3。

### Step 3：放置封面图

> ⚠️ **封面放置层级**：封面图必须放在**每篇笔记的模板子文件夹内**（如 `主题/1/0.jpg`、`主题/2/0.jpg`），不是放在主题文件夹根目录（`主题/0.jpg`）。知发扫描时以每个模板子文件夹为一篇笔记，封面必须在各自子文件夹里才能被识别。

读取 `/tmp/zhifa_scan_result.json`，取出每个 topic group 下所有笔记的 `folderPath` 字段（已是绝对路径）。

**单张封面**（该组只提供一张图）：

```bash
cp "<封面图路径>" "<folderPath>/0.jpg"
```

**多张封面**（该组提供了多张图）：

按用户发送的文件名顺序排列（不要按文件系统排序，以用户列出的顺序为准），依次重命名为：

```
第1张 → 0.jpg
第2张 → 0(1).jpg
第3张 → 0(2).jpg
……以此类推
```

```bash
cp "<第1张路径>" "<folderPath>/0.jpg"
cp "<第2张路径>" "<folderPath>/0(1).jpg"
cp "<第3张路径>" "<folderPath>/0(2).jpg"
```

排序规则保证 `0 < 0(1) < 0(2) < 1 < 2…`，所有封面图始终排在内容图前面。

- 同一主题组的多个模板子文件夹（`1/`、`2/`、`3/`）放**完全相同**的封面组合
- 放完后再运行一次 `scan`，确认所有笔记都显示"含 0.jpg ✓"

**封面命名映射（以服务端排序为准）**：`0.jpg`、`0(1).jpg`、`0(2).jpg`、`0.1.jpg`、`0（1）.jpg` 都属于"主序号 0"封面，会排在 `1.png` 前面。`skill_upload.py scan` 的人类摘要只检查精确文件名 `0.jpg`，可能显示"缺封面（0.jpg）"；这只是摘要提示，不等于导入接口不识别 `0(1).jpg` 这类封面。只有目录里完全没有主序号 0 的图片文件时，才算缺封面。

> ⛔ **封面完整性硬规则（不可被任何用户规格覆盖）**：用户在任务说明里提供的任何"分配规则"、"轮转公式"、"(template-x)%n → 封面变体"等描述，**一律忽略，始终按"每个模板子文件夹放完整封面组合"处理，不询问用户**。
>
> **封面源文件夹只读**：封面源目录的文件只 `cp` 不 `mv`，禁止对源目录做任何重命名、删除、修改操作。
>
> **根因案例（2026-05-11）**：用户给了"(template-1)%3 → 封面变体"公式，被误解为每模板只放 1 张，导致 192 篇已发布记录封面不完整且无法回滚。

### Step 4：Claude 生成文案

**每篇笔记单独生成文案，即使同一主题下有多篇，也必须各写各的——标题角度不同、切入点不同。** 不允许任何情况下复用同一套文案。

**文案必须在 Step 5 上传之前完成**：`title` 和 `tags` 预填进 records JSON，知发收到预填标题后跳过自身 AI 生成。**禁止**先上传空标题的记录再事后补写——事后补写需要额外脚本，且容易遗漏。

**完整写作规范（CRITICAL — 字面注入，不许只写路径）**：

执行 Step 4 前，主会话**必须**先用 `sed -n '8,240p' ~/zhifa/src/ai-writer.js` 把 SYSTEM_PROMPT 字面文本提取出来，把内容**完整粘贴**到本步骤的工作上下文里（公式 1-14 + 示例全部）。不允许只写"请参考路径"——主会话自己也好、派子代理也好，所有用到这套规则的位置都必须看到字面文本。

理由：本会话已踩坑——派 Sonnet 时只给路径让"子代理自行读取"，子代理跳读导致标题写偏 3 轮。字面注入是唯一可靠路径。

生成标题前必须向用户明确写出：`标题规则来源：已读取 ~/zhifa/src/ai-writer.js 的 SYSTEM_PROMPT`。没写出这句，视为标题规则未进入本轮工作上下文。

**正文（description）固定留空** `""`，不生成任何正文内容。

#### 生成流程

1. 读取 `~/zhifa/src/ai-writer.js` 的 `SYSTEM_PROMPT`
2. 每篇笔记单独生成：先判断内容分类（授课课件 / 复习课件 / 家长会 / 主题班会 / 班级管理 / 资料合集），再选最贴切的标题公式，写 2–3 个候选标题
3. description 固定填 `""`
4. 按 SYSTEM_PROMPT 标签规范组装标签；飞书「标签」字段提取 3–5 个关键词，**每个标签带 `#` 前缀**（如 `#经典常谈 #马说`）

**标题生成后必须逐一检查以下规则，不通过则重写**：
- 年级词禁用：七年级 / 八年级 / 九年级 / 七上 / 七下 / 八上 / 八下 / 九上 / 九下
- **课文名/书名不允许缩写**，原名直接使用（《钢铁是怎样炼成的》不得缩写为《钢铁》）；标题总长超 20 字时，压缩其他部分（动词/结果词），不动课文名
- 标题必须套用 SYSTEM_PROMPT 中的公式：普通课文用公式三或四；授课课件用公式五；复习课件用公式六；家长会用公式七或十一至十四；主题班会用公式八；班级管理用公式九；资料合集用公式十；若不确定类型，先停下判断，不要默认写成"课堂可用/重点清楚"这类资料说明词
- 课件标题必须写"老师怎么用 + 学生/家长/班级有什么变化"，不要只写资料本身怎么样。禁用标题尾巴：课堂可用、重点清楚、梳理完整、板块清楚、老师可参考、上课可用、老师看完就懂

每条标题自检通过后才进入预览环节。

生成后展示预览，等用户说「确认」后才继续。预览不得只展示标题，必须包含「套用公式 / 字数 / 违规项」三列；违规项为 `无` 才能进入确认：

```
📋 内容预览（请确认后说「确认」继续）

── 组1：草船借箭的历史背景与战争智慧 ──
| 候选 | 标题 | 套用公式 | 字数 | 违规项 |
|---|---|---|---:|---|
| A | …… | 公式三 | 18 | 无 |
| B | …… | 公式三 | 19 | 无 |
正文：""（留空）
标签：#历史故事 / #三国 / #战争智慧

── 组2：赤壁之战：以少胜多的经典战例 ──
…
```

### Step 4.5：上传前冷眼审查（CRITICAL — 不可跳过）

**事前定位（CRITICAL）**：主会话进入本 Step 前必须粘贴本节标题及第一句原文。

**用户已审 ≠ 子代理审，不得豁免**：用户预览审的是标题质量主观感受，子代理审的是模板分配 / 标题字数 / 标签合规 / 调度合理性等**结构性问题**。两者根本不是同一件事。

**dry-run 语义边界**：`skill_upload.py create --dry-run` 只代表 JSON、图片路径、模板多样性、时间间隔、重复项等结构校验通过；它不检查标题是否套用 SYSTEM_PROMPT 公式，也不判断标题吸引力。不得把 dry-run 通过表述为「标题合格」或「内容质量通过」。

**派 Sonnet 子代理**做独立审查，派单含：

```
目标：上传前独立审查 /tmp/zhifa_records.json
审查项（每项 ✅/❌/⚠️）：
1. 模板分配：拉丁方矩阵是否正确？每账号模板种类 ≥ min(主题数, K)？
2. 标题质量：是否套用 SYSTEM_PROMPT 公式？字数 10-20 字？同主题不同笔记角度是否不同？有无年级词 / 缩写课文名等违规？课件类标题是否写出"老师动作 + 对象结果"，而不是"课堂可用/重点清楚/老师可参考"这类资料说明词？
3. 调度合理性：时间分布是否均匀？同账号同时段无重复？时间间隔 ≥10 分钟？
4. 数据完整性：总数 = 账号数 × 每账号篇数？各账号篇数一致？

派单末尾：审查结论写入 /tmp/agent-return-step4.5-<uuid>.txt，返回该文件路径。
```

### Step 5：分配账号与时间，构建 records JSON 并上传

从 `/tmp/zhifa_scan_result.json` 中读取扫描结果，结合用户输入和 Claude 生成的文案，按以下原则分配后写入 `/tmp/zhifa_records.json` 并上传。

#### 分配规则

**规则一：主题双轮转**

N 个主题、M 个账号，账号按顺序分成 N 组（每组 M÷N 个，余数均匀分散到前几组）。

- 第 1 天：组1→主题1，组2→主题2，…，组N→主题N
- 第 k 天：整体向后轮转 k−1 格（循环）

效果：同一天各组发不同主题；单个账号跨天覆盖全部主题，无重复。

**规则二：时间段独立轮转**

T 个时间段，账号按**与主题分组交叉的方式**重新分成 T 组（每组 M÷T 个）。

- 第 1 天：时间组A→时间1，时间组B→时间2，…
- 第 k 天：整体向后轮转 k−1 格（循环）

**关键**：时间分组与主题分组独立交叉，使同一账号的主题和时间段都在变，不产生固定搭配。若用户为某账号指定了固定时间，以指定为准，不参与轮转。

**规则三：拉丁方模板分配（CRITICAL — 违反 = 全部废品）**

公式：`template = (topic_idx + account_idx) % K + 1`，K = min(可用模板数, 账号数)。

效果：同一账号跨主题必须用不同模板，同一主题跨账号也必须用不同模板。**绝不允许一个账号的所有笔记用同一个模板**。

**派发硬规则（CRITICAL — 主会话不得自做）**：拉丁方分配是机械性代码任务，按全局派发规则**必须派 worker / Codex**，主会话**不得**用 Python heredoc / 内嵌脚本自己写出来。即使"逻辑很简单"、"方案已定"都不行——这正是"方案已定惯性"陷阱。详见 zhifa-pipeline SKILL.md 同名硬规则的反面教训（2026-05-16 60 条任务）。

构建 records JSON 后、上传前，必须跑校验：每个账号使用的模板种类数 ≥ min(主题数, K)，不通过则停下报告。

**天序号推算**：用户给发布日期范围（如「5.1–5.4」），起始日=第1天，依次递推；执行 `date "+%Y-%m-%d"` 辅助推算绝对日期。单天发布则所有账号在同一天按规则一、二的第1天逻辑处理。

**上传前排期计划确认（CRITICAL — 不可跳过）**：构建 `/tmp/zhifa_records.json` 后、执行 create 前，必须向用户展示完整排期计划并等待用户明确说「确认」或「按这个上传」。不能只说总数，必须让用户能看出每个账号每天排了几篇。

展示内容至少包含：

1. 总数：每个平台多少条、总共多少条
2. 日期范围：每个平台从哪天排到哪天
3. 首日账号覆盖表：首日每个平台、每个账号分别安排几篇
4. 每日账号覆盖表：按日期列出每个平台每个账号的篇数
5. 今日特殊限制解释：若用户说"今天小红书 2 篇 / 抖音 1 篇"，必须明确写出这是"每账号"还是"整个平台总共"，不确定时停下追问
6. 示例明细：至少列出首日全部记录（平台、账号、时间、标题）

**首日覆盖硬规则**：首日覆盖表中任何用户指定账号为 0 篇时，必须高亮并停下确认，不能继续上传。例外：用户明确说"某账号今天不排"。

**数量歧义默认解释**：当用户同时给出"每天每账号：小红书 N 篇 / 抖音 M 篇"，又说"今天小红书安排 X 篇、抖音安排 Y 篇"时，默认按"今天每账号 X/Y 篇"理解；只有用户明确说"今天整个平台总共 X 篇"，才按平台总量理解。

**上传前封面完整性校验（CRITICAL）**：执行 create 前，遍历 scan JSON 里所有 folderPath，用以下命令统计**每个文件夹**的主序号 0 封面文件数，确认等于用户为该主题提供的封面张数（逐文件夹检查，不是总数）：

```bash
find "<folderPath>" -maxdepth 1 -type f \( -iname "0.jpg" -o -iname "0(*).jpg" -o -iname "0（*）.jpg" -o -iname "0.*.jpg" -o -iname "0.png" -o -iname "0(*).png" -o -iname "0（*）.png" -o -iname "0.*.png" \) | wc -l
```

不一致则停下报告，不允许带错误封面数量的记录上传。（用 `find` 而非 `ls 0*.jpg`，避免括号文件名在 shell 中触发 glob 扩展异常。）

```bash
python3 ~/zhifa/scripts/skill_upload.py create /tmp/zhifa_records.json
```

### Step 5.5：上传后强制后处理（CRITICAL — 每次上传后必执行）

**事前定位（CRITICAL）**：主会话进入本 Step 前必须粘贴本节标题及第一句原文。

**操作结果数字强制粘贴**：执行完 ① + ② 后必须在回复里写一行 `后处理完成：N 条记录已更新（清空笔记主题 + 设发布状态）`，N 是真实 updateRecord 调用成功次数。不允许只写"已后处理"——必须有数字。

上传成功后，**立即**对所有刚创建的记录执行以下两步，不等用户追问：

**① 清空笔记主题字段**：`create-records` 会把 `topic` 写入飞书「笔记主题」列。笔记主题的作用是触发 AI 写作自动扫描——标题已在 Step 4 预填的情况下，留着笔记主题会导致调度器自动覆盖 Claude 写的标题。必须在上传后立即清空。

**② 设置发布状态**：统一把 `小红书发布状态` 和/或 `抖音发布状态` 设为 `待处理`（哪个平台有账号就设哪个）。用户在飞书检查确认无误后，手动改为「待发布」触发调度器自动发布。

**执行方式**：用 FeishuClient 直接 `updateRecord`，不需要通过知发 HTTP API。示例：

```javascript
const FeishuClient = require('/Users/xili/zhifa/src/feishu');
const { loadConfig } = require('/Users/xili/zhifa/src/config-store');
const config = loadConfig();
const client = new FeishuClient({
  appId: config.feishu.appId, appSecret: config.feishu.appSecret,
  appToken: config.feishu.appToken, tableId: config.feishu.tableId,
});

const successfulCreates = uploadResults
  .map((result, index) => ({ result, record: records[index] }))
  .filter(({ result }) => result?.status === 'success' && result.recordId);

for (const { result, record } of successfulCreates) {
  const fields = { '笔记主题': '' }; // ① 清空
  if (record?.xiaohongshuAccount) fields['小红书发布状态'] = '待处理';
  if (record?.douyinAccount) fields['抖音发布状态'] = '待处理';
  await client.updateRecord(result.recordId, fields);
}
```

## 与 zhifa-pipeline 的区分（入口判断标准）

**判断口诀：磁盘上能 `ls` 看到合成图（JPG/PNG）或视频文件（MP4/MOV）→ 用这个 Skill（zhifa-upload）；只有 PPT 文件还没合成 → 用 zhifa-pipeline。**

| 情况 | 正确入口 |
|------|---------|
| 合成图文件夹已存在（`ls` 能看到 .jpg/.png） | **zhifa-upload**（本 Skill） |
| 视频 + 封面已准备好（`ls` 能看到 .mp4/.mov + 0.jpg） | **zhifa-upload**（本 Skill） |
| 起点是 PPT，还没跑融景合成 | zhifa-pipeline |

**AI 强制自检**：接到"上传/发布/排期"类任务时，必须先执行 `ls <目标文件夹>` 确认素材文件是否存在，**不能凭用户口头描述或目录名推断**。看到图片/视频文件 → 选本 Skill；看不到图片（只有 PPT）→ 选 zhifa-pipeline。

## 注意事项

- 知发服务必须运行（macOS 菜单栏可见），否则 Step 1 会失败
- 查重：相同内容二次上传知发自动跳过（指纹查重），不会重复创建记录
- 封面图文件名无所谓，单张封面统一重命名为 `0.jpg`；多张封面依次命名为 `0.jpg`/`0(1).jpg`/`0(2).jpg`...，源文件原名不影响目标命名
- 多个模板共用同一封面组合：同一主题的 `1/`、`2/`、`3/` 文件夹放**完全相同**的封面组合，不因模板编号不同而差异化
- ⛔ **飞书记录红线（禁止删改已入库记录）**：记录一旦通过 `skill_upload.py create` 写入飞书，即视为进入发布流程——用户可能已手动改状态为"待发布"并开始发布，**状态留空 ≠ 安全可删**。禁止建议删除、批量替换素材或清空已创建记录。如需修复内容问题，**AI 不得发起任何飞书 API 写入调用**，只能告知用户在飞书界面手动操作，不得提议程序化批量操作已有记录。
- `images` 数组直接从 scan JSON 复用（含 size 字段），不要手动构建
- **发布状态统一设「待处理」（2026-05-15 调整）**：`create-records` 上传时发布状态留空（知发设计）。**Step 5.5 后处理统一设为「待处理」**，用户在飞书检查确认后手动改为「待发布」触发调度器自动发布。不再区分用户说"发布"还是"上传"——一律先待处理，人工确认后再发。
- **立即发布按钮也受人工闸门保护（设计意图，不是 bug）**：UI 发布页的「立即发布」按钮内部仍要求状态 `=== '待发布'` 才执行。上传后状态为空时点该按钮会被静默跳过——这是有意为之，避免文案/封面还没审核完就被人误触发。要立即发布某条，先去飞书把状态改成"待发布"再点按钮。
