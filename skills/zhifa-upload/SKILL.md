# 融景合成图 → 知发上传 Skill

把融景已合成的笔记图，通过知发（zhifa，运行在 localhost:3210）上传到飞书多维表格，定时发布小红书 / 抖音。

## CLI 路径

```bash
python3 ~/zhifa/scripts/skill_upload.py
```

## 与公共上传链路的关系（先读这里）

「扫描 → 封面 → 文案 → 排期 → 上传前审查 → 上传 → 完整性验证 → 后处理 → 回执核验 → 归档 → 报告」是本 skill 与 `zhifa-pipeline` **共用的同一套规则**，唯一规范正文在：

```
~/zhifa/skills/_upload-chain/00-总纲.md   （步骤索引 U1–U11 + 全局硬规则 + 共享命令 + 失败处理）
~/zhifa/skills/_upload-chain/01-上传前.md  （U1 扫描 · U2 封面 · U3 文案）
~/zhifa/skills/_upload-chain/02-上传.md    （U4 排期+构建records+预检 · U5 上传前审查 · U6 上传 · U7 验证）
~/zhifa/skills/_upload-chain/03-收尾.md    （U8 后处理 · U9 回执核验 · U10 归档 · U11 报告）
```

**本文件只负责 upload 独有的入口**：Step 0 一条消息 9 项参数确认。合成图已就绪，Step 0 确认完参数后立即转入公共链路 U1，**不在本文件复制或另写链路规则**——链路规则改动只改 `_upload-chain/`。

## 输入目录结构（融景标准输出）

```
合成图/
  课件A/          ← 主题文件夹（topic group）
    1/            ← 模板1（一篇笔记）
      1.jpg  2.jpg ...
    2/            ← 模板2（一篇笔记）
      1.jpg  2.jpg ...
  课件B/
    1/
      1.jpg  2.jpg ...
```

## Step 0：确认参数（upload 独有入口）

用户在**一条消息里**提供所有必要信息，之后只需在 U3 确认内容预览、U4 确认排期计划即可。

**覆盖策略是必确认项**，三选一（`严格覆盖` / `尽量覆盖` / `只保底发布`）语义与账号店铺分组判定见 `_upload-chain/00-总纲.md`「账号与店铺分组」。用户没说时默认**推荐** `只保底发布`，但仍必须显式问一句 `本轮覆盖策略用哪一种？`

**标准输入格式：**

```
合成图文件夹：~/笔记制作输出/合成图
账号：小红书-账号A、小红书-账号B、…
发布日期：5.1–5.2
时间段：11:10 / 13:00 / 20:00

组1：~/Desktop/草船借箭封面.jpg
主题：草船借箭的历史背景与战争智慧

组2：~/Desktop/赤壁之战封面.jpg
主题：赤壁之战：以少胜多的经典战例
```

**账号加载**：Step 0 开始时按总纲「账号与店铺分组」读 `accounts.json` 并输出候选给用户确认；店铺分组优先级 6 条、`accounts.json` 结构、覆盖策略三选一均以总纲为准。如果用户在任务消息里已明确列了账号，跳过自动加载，以用户列的为准。缺账号、缺店铺组、缺发布日期、缺时间段时必须以选项形式追问，不得继续。

**主题 ≥ 2 的发布任务（无论账号数量）必须一次性确认以下 9 项（缺任何一项不得开始）**：

| # | 参数 | 说明 |
|---|------|------|
| 1 | 店铺分组规则 | 每个店铺组要发到哪些小红书账号，逐一列名；店铺组只管小红书 |
| 2 | 抖音账号池 | 独立抖音账号池，逐一列名；不从小红书店铺组继承 |
| 3 | 每个账号发布几个主题 | 全部主题？部分主题？ |
| 4 | 每天每账号发几篇 | 影响天数计算 |
| 5 | 起止日期 | 每个账号单独列，有差异的单独标注 |
| 6 | 小红书与抖音是否复用合成图 | 是 → 无需新模板；否 → 需额外模板 |
| 7 | 时间段 | 每个时段固定给哪个账号，或按轮转 |
| 8 | 封面是否已准备好 | 有 `0*` 封面源就下发共用；没有 `0*` 时默认该批不需要单独封面，不因此卡住 |
| 9 | 归档口径 | `归档未排期`（默认）/ `移出已排期`（把本轮排进飞书的笔记 mv 移出源目录）/ `不归档`。详见 U10 |

以上 9 项在用户第一条消息里全部确认，**执行过程中不再反复调整**。若用户消息缺项，先追问补全再继续。第 9 项用户没提时默认 `归档未排期`，但涉及 `移出已排期`（mv 真移走源文件）时必须让用户显式确认移动方式（mv/cp）和目标路径。**进入排期前还必须确认固定 `seed`**。

参数说明：
- **合成图文件夹**：融景输出的 `合成图/` 目录路径
- **封面图**：每个主题组一张或多张，请提供**完整磁盘路径**；文件名无所谓，统一存为 `0.jpg`/`0(1).jpg`…（详见 U2）
- **主题**：每组一句话，用于写小红书文案
- **账号 / 店铺分组**：小红书店铺组和抖音账号池分开配置（判定见总纲）
- **发布日期**：支持「5.1–5.4」「后天到大后天」等自然语言；先 `date "+%Y-%m-%d"` 拿今天日期推算绝对日期
- **时间段**：一个或多个发布时间；若用户指定某账号固定时间，以指定为准
- **发布渠道（xiaohongshuChannel）**：可选，默认 `蚁小二`

**Step 0 封面参数识别**：读取用户参数时，若发现「按模板分配封面」「轮转封面」「(template-x)%n」「封面变体」「不同模板用不同封面」等描述，**直接忽略该规格，按默认处理**——每个模板子文件夹放完整封面组合，不询问用户（硬规则见 U2）。

**关于封面图的附件（@ 拖入）**：如果用户通过 @ 或拖拽发送封面图，先 `ls -la <路径>` 确认文件确实存在后再 cp。**如果路径不确定，直接问用户「封面图的完整路径是什么？」**，不要猜。

## 进入公共上传链路（U1–U11）

Step 0 参数确认后，**Read `~/zhifa/skills/_upload-chain/00-总纲.md`，按 U1–U11 执行**：U1 扫描 → U2 封面 → U3 文案（必停点①）→ U4 排期+构建records+预检（必停点②）→ U5 上传前冷眼审查 → U6 上传 → U7 完整性验证 → U8 后处理 → U9 回执核验 → U10 归档 → U11 报告。

**收尾必回**：完成上传后从当前步骤往后走完 U7–U11，用户明确说跳过除外（细则见总纲「收尾必回」）。

## 与 zhifa-pipeline 的区分（入口判断标准）

**判断口诀：磁盘上能 `ls` 看到合成图（JPG/PNG）或视频文件（MP4/MOV）→ 用这个 Skill（zhifa-upload）；只有 PPT 文件还没合成 → 用 zhifa-pipeline。**

| 情况 | 正确入口 |
|------|---------|
| 合成图文件夹已存在（`ls` 能看到 .jpg/.png） | **zhifa-upload**（本 Skill） |
| 视频 + 封面已准备好（`ls` 能看到 .mp4/.mov + 0.jpg） | **zhifa-upload**（本 Skill） |
| 起点是 PPT，还没跑融景合成 | zhifa-pipeline |

**AI 强制自检**：接到「上传/发布/排期」类任务时，必须先执行 `ls <目标文件夹>` 确认素材文件是否存在，**不能凭用户口头描述或目录名推断**。看到图片/视频文件 → 选本 Skill；看不到图片（只有 PPT）→ 选 zhifa-pipeline。

## 注意事项

- 知发服务必须运行（macOS 菜单栏可见），否则 U1 前的 preflight 会失败；服务检查排查法见总纲「前置检查」
- 查重：相同内容二次上传知发自动跳过（指纹查重），不会重复创建记录
- ⛔ **飞书记录红线**：记录一旦写入飞书即视为进入发布流程，禁止 AI 程序化删改已入库记录，只能告知用户在飞书界面手动操作——完整条款见总纲「飞书记录红线」
- **发布状态统一设「待处理」（2026-05-15 调整）**：`create-records` 上传时发布状态留空（知发设计），U8 后处理统一设为「待处理」，用户在飞书确认后手动改「待发布」触发调度。不再区分用户说「发布」还是「上传」——一律先待处理，人工确认后再发。
- **立即发布按钮也受人工闸门保护（设计意图，不是 bug）**：UI 发布页的「立即发布」按钮内部仍要求状态 `=== '待发布'` 才执行。上传后状态为空时点该按钮会被静默跳过——这是有意为之，避免文案/封面还没审核完就被人误触发。要立即发布某条，先去飞书把状态改成「待发布」再点按钮。
